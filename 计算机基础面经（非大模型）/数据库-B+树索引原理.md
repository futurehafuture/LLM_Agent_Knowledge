# B+ 树索引原理

## 1. 面试官到底想考什么

这题是 MySQL 索引题的深水区。面试官不是只想听“B+ 树查询快”，而是想看你是否理解数据库索引为什么要适配磁盘。

核心考点是：**B+ 树为什么树高低、为什么磁盘 IO 少、为什么适合范围查询、B+ 树和 B 树有什么区别**。

## 2. 零基础版解释

想象你在图书馆找一本书。

如果书架是一条长长的队伍，你只能一本本找，太慢。

如果书架按分类分层：一楼是大类，二楼是小类，最后一层才是具体书。你每次只需要看很少几个目录，就能快速定位。

B+ 树就像这种多层目录。上面的节点主要负责指路，真正的数据都放在最底层。最底层的数据还按顺序连在一起，所以如果你要找“价格 100 到 200 的订单”，找到第一个之后可以顺着链表往后扫，不用反复从根节点查。

## 3. 正式概念

**B+ 树** 是一种多路平衡搜索树。MySQL InnoDB 常用 B+ 树作为索引结构。

它的主要特点：

- **多路分叉**：一个节点可以存很多 key，所以树高很低。
- **所有数据都在叶子节点**：非叶子节点主要存 key 和指针，用来导航。
- **叶子节点有序并通过链表相连**：适合范围查询和排序扫描。
- **平衡结构**：从根节点到任意叶子节点的路径长度相同，查询稳定。

B 树和 B+ 树的区别：

| 对比项 | B 树 | B+ 树 |
|---|---|---|
| 数据存放位置 | 非叶子节点和叶子节点都可以存数据 | 数据都在叶子节点 |
| 非叶子节点作用 | 既导航也存数据 | 主要导航 |
| 范围查询 | 需要中序遍历，较麻烦 | 叶子节点链表顺序扫描，效率高 |
| 单个节点可存 key 数 | 因为要存数据，key 数相对少 | 非叶子节点不存整行数据，key 数更多 |
| 树高 | 相对可能更高 | 通常更矮 |

为什么索引用 B+ 树不用 B 树：

- 数据库查询常常受磁盘 IO 限制，B+ 树非叶子节点能放更多 key，树更矮，IO 次数更少。
- B+ 树所有数据都在叶子节点，查询路径长度更稳定。
- B+ 树叶子节点有序链表，特别适合范围查询、排序和分页扫描。
- B+ 树更适合数据库“按块读取”的存储方式。

### 3.1 为什么数据库索引要考虑磁盘 IO

数据库里的数据通常不可能全部放在内存中，尤其是大表。很多数据存放在磁盘上。磁盘读取比内存访问慢得多，所以数据库索引设计的核心目标之一是：**尽量减少磁盘 IO 次数**。

InnoDB 以页为单位读取数据，常见页大小是 16KB。也就是说，数据库不是每次只读一个字段，而是一次读一页。

如果索引结构能让每一页存更多 key，就能减少树的高度。树越矮，从根节点走到叶子节点需要读的页越少。

### 3.2 为什么不是二叉搜索树

二叉搜索树每个节点最多只有两个孩子。数据量很大时，树高可能很高。

假设有 1000 万条数据：

- 平衡二叉树高度大约是 `log2(10000000)`，约 24 层。
- B+ 树每个节点可以有几百甚至上千个分叉，高度可能只有 3 到 4 层。

如果每访问一层都可能触发一次磁盘 IO，那么 24 次和 3 次差距非常大。

所以数据库索引不用普通二叉树，核心原因不是二叉树不能查，而是它不适合磁盘页读取模型。

### 3.3 B+ 树节点里大概存什么

可以简化理解：

非叶子节点：

```text
[key1 | pointer1] [key2 | pointer2] [key3 | pointer3] ...
```

它主要负责导航，不保存整行数据。

叶子节点：

```text
[key -> row data] [key -> row data] [key -> row data] ...
```

对于 InnoDB 主键索引，叶子节点保存整行数据。

对于二级索引，叶子节点通常保存：

```text
二级索引列值 -> 主键值
```

### 3.4 一次 B+ 树查询过程

假设要查：

```sql
SELECT * FROM user WHERE id = 100;
```

主键索引查询过程：

1. 从根节点开始，判断 `100` 应该落在哪个子节点范围。
2. 进入下一层内部节点，继续判断范围。
3. 最终到达叶子节点。
4. 在叶子节点中找到 `id = 100` 的整行数据。

由于 B+ 树高度低，这个过程通常只需要少量页访问。

### 3.5 范围查询为什么快

例如：

```sql
SELECT * FROM order_info
WHERE id BETWEEN 100 AND 200;
```

B+ 树会先定位到 `id = 100` 所在的叶子节点，然后沿着叶子节点链表向后扫描，直到超过 200。

这比每个 id 都从根节点重新查一遍更高效。

这也是 B+ 树比哈希索引更适合数据库通用索引的重要原因：哈希索引适合等值查询，但天然不适合范围查询。

### 3.6 B+ 树和哈希索引的对比

| 对比项 | B+ 树索引 | 哈希索引 |
|---|---|---|
| 等值查询 | 快 | 很快 |
| 范围查询 | 支持 | 不适合 |
| 排序 | 支持 | 不支持 |
| 前缀匹配 | 联合索引可支持 | 不支持 |
| 磁盘场景 | 适合 | 场景有限 |

MySQL 也不是完全不用哈希思想。例如某些存储引擎或自适应哈希索引会使用哈希优化，但通用索引结构仍以 B+ 树为主。

### 3.7 主键为什么建议短且递增

InnoDB 的主键索引是聚簇索引，主键值会出现在二级索引叶子节点中。

所以主键太长会导致：

- 主键索引本身变大。
- 所有二级索引也变大。
- 缓存命中率下降。

主键递增的好处：

- 新数据通常追加到索引末尾。
- 减少页分裂。
- 写入更顺序，性能更稳定。

如果使用完全随机 UUID 作为主键，可能导致数据插入位置分散，引发更多页分裂和随机 IO。

## 4. 面试回答模板

MySQL InnoDB 常用 B+ 树作为索引结构，主要是因为它适合磁盘存储和范围查询。

B+ 树是一种多路平衡搜索树，一个节点可以存很多 key，所以树的高度比较低。数据库查询索引时通常要从磁盘读取页，树越矮，需要的磁盘 IO 次数就越少。

B+ 树和 B 树的主要区别是：B 树的非叶子节点和叶子节点都可以存数据，而 B+ 树只有叶子节点存真正的数据，非叶子节点主要用于导航。这样 B+ 树的非叶子节点可以放更多 key，分叉更多，树更矮。

另外，B+ 树的叶子节点之间是有序链表连接的，所以范围查询很方便。比如查 id 在 100 到 200 之间的数据，只要先定位到 100，然后沿着叶子节点链表往后扫即可。

所以 MySQL 索引用 B+ 树，核心原因是它能减少磁盘 IO，查询稳定，并且非常适合范围查询。

## 6. 常见误区

- **误区 1：只说 B+ 树查询快。** 要解释快在哪里：树高低、IO 少、范围查询方便。
- **误区 2：不知道数据是否存在非叶子节点。** B+ 树真正数据在叶子节点，非叶子节点主要导航。
- **误区 3：把 B+ 树和二叉搜索树混淆。** B+ 树是多路树，不是每个节点只有两个孩子。
- **误区 4：忽略磁盘 IO。** 数据库索引设计首先要考虑磁盘页读取成本。
- **误区 5：认为哈希索引一定更好。** 哈希适合等值查询，但不适合范围查询和排序。

## 7. 记忆口诀

**上层只指路，数据在叶子；叶子连成链，范围查询快。**

B+ 树是一种多路平衡搜索树，InnoDB 常用它做索引。它的非叶子节点主要存 key 和指针，真正数据在叶子节点，叶子节点按顺序链表连接。相比 B 树，B+ 树非叶子节点能放更多 key，树更矮，磁盘 IO 更少，而且范围查询可以沿叶子链表顺序扫描，所以更适合数据库索引。
