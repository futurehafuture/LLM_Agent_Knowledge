# TCP 如何保证可靠性

## 1. 面试官到底想考什么

面试官不是想听一句“TCP 是可靠的”，而是想看你是否理解：网络传输本来是不可靠的，数据包可能丢、乱序、重复、损坏、传太快。TCP 要做的事，就是在这些不可靠条件下，尽量让应用层看到一条可靠、有序的字节流。

这题也常用来判断你是不是只背过八股。只说“三次握手”通常会显得比较浅，因为三次握手主要负责建立连接，不是传输过程中保证可靠的全部机制。

## 2. 零基础版解释

可以把 TCP 想成“寄快递但每个包裹都要确认签收”。

你要给别人寄一本书，但书被拆成很多页分别寄出。可能发生这些情况：

- 有一页丢了。
- 第 5 页比第 3 页先到。
- 同一页被重复送了两次。
- 某一页在路上被弄坏了。
- 你寄得太快，对方来不及收。
- 整个快递网络太堵，继续猛寄只会更糟。

TCP 的做法是：每一页编号，对方收到后告诉你“我收到哪一页了”；如果等太久没确认，你就重寄；对方按编号重新排好顺序；如果对方收不过来，你就慢一点；如果网络拥堵，你也要降速。

## 3. 正式概念

TCP 通过一组机制提供可靠传输：

- **序列号**：给每段数据编号，用于识别顺序、去重和重组。
- **确认应答**：接收方告诉发送方已经收到哪些数据。
- **超时重传**：发送方在一定时间内没收到确认，就重新发送。
- **快速重传**：连续收到重复确认时，发送方推测某段丢失，提前重传。
- **校验和**：检测数据在传输中是否损坏。
- **滑动窗口**：允许连续发送多个数据段，提高效率。
- **流量控制**：根据接收方处理能力调整发送速度，避免把接收方压垮。
- **拥塞控制**：根据网络拥堵程度调整发送速度，避免把网络压垮。
- **连接管理**：通过三次握手建立连接，通过四次挥手释放连接。

### 3.1 TCP 面向连接是什么意思

TCP 是**面向连接**的协议。这里的“连接”不是一根真实的线，而是通信双方在内存中维护的一组状态。

建立连接后，客户端和服务端都会记录：

- 对方的 IP 和端口。
- 当前发送到哪个序列号。
- 当前确认到哪个序列号。
- 接收窗口还有多大。
- 拥塞窗口大概能发多少。
- 连接处于什么状态，例如 `ESTABLISHED`、`FIN_WAIT`、`TIME_WAIT`。

所以 TCP 的连接，本质上是双方共同维护的一套通信上下文。只要这套状态存在，双方就能基于它进行可靠传输。

### 3.2 序列号和确认应答

TCP 传输的是**字节流**，不是一条条独立消息。为了知道哪些字节已经传到了，TCP 会给字节编号。

假设发送方发送了 1000 个字节，序列号从 100 开始：

```text
发送数据：序列号 100 到 1099
接收方返回 ACK = 1100
含义：1100 之前的字节我都收到了，下次请从 1100 开始发
```

TCP 的 ACK 通常是**累计确认**。也就是说，ACK = 1100 表示 1100 之前的数据都已经按序收到。

这个机制解决了三个问题：

- **丢包识别**：一直没收到确认，说明可能丢了。
- **乱序重排**：接收方可以按序列号重新排序。
- **重复去重**：同一段数据重复到达时，可以根据序列号丢弃重复数据。

### 3.3 超时重传和快速重传

网络中数据包可能丢失。TCP 的基本处理方式是：如果发送方迟迟等不到 ACK，就重传。

**超时重传**：

- 发送数据后启动计时器。
- 超过一定时间还没收到 ACK，就认为数据可能丢失。
- 重新发送这段数据。

**快速重传**：

- 如果接收方发现中间缺了一段数据，会反复确认自己已经连续收到的位置。
- 发送方如果连续收到多个重复 ACK，就推测中间某段丢了。
- 不等超时，直接重传丢失的数据段。

例如：

```text
发送方发送：1, 2, 3, 4, 5
接收方收到：1, 2, 4, 5
接收方会反复说：我期待 3
发送方收到多个重复 ACK 后，快速重传 3
```

### 3.4 滑动窗口为什么能提高效率

如果 TCP 每发送一段数据就必须等确认，再发下一段，效率会很低。尤其网络延迟较高时，大量时间都浪费在等待上。

**滑动窗口**允许发送方在未收到确认前，连续发送多个数据段。

可以理解为：

```text
已经确认的数据 | 已发送未确认的数据 | 还能继续发送的数据 | 暂时不能发送的数据
```

随着 ACK 返回，窗口向右滑动，发送方就可以继续发送新的数据。

滑动窗口解决的是效率问题：既保持可靠性，又不用“一发一等”。

### 3.5 流量控制和拥塞控制的区别

这两个概念很容易混。

| 概念 | 控制对象 | 目的 | 依据 |
|---|---|---|---|
| 流量控制 | 接收方 | 避免接收方处理不过来 | 接收窗口 `rwnd` |
| 拥塞控制 | 网络链路 | 避免网络被打爆 | 拥塞窗口 `cwnd` |

**流量控制**看的是“对方还能不能收”。接收方会告诉发送方自己的接收窗口还有多大。如果接收方缓存快满了，就把窗口调小，让发送方慢一点。

**拥塞控制**看的是“网络还能不能扛”。如果网络拥堵，丢包变多，TCP 会降低发送速度。常见机制包括慢启动、拥塞避免、快重传、快恢复。

实际发送窗口通常受两者共同限制：

```text
发送窗口 = min(接收窗口 rwnd, 拥塞窗口 cwnd)
```

### 3.6 三次握手和可靠性有什么关系

三次握手主要解决连接建立问题，不是传输可靠性的全部。

三次握手做了几件事：

- 确认双方发送和接收能力正常。
- 交换初始序列号。
- 建立连接状态。

简化过程：

```text
1. 客户端：SYN，我想建立连接，我的初始序列号是 x
2. 服务端：SYN + ACK，我同意连接，我的初始序列号是 y，我确认了 x
3. 客户端：ACK，我确认了 y
```

之后真正的数据传输可靠性，主要靠序列号、ACK、重传、窗口和控制机制。

### 3.7 TCP 和 UDP 的基础对比

| 对比项 | TCP | UDP |
|---|---|---|
| 是否连接 | 面向连接 | 无连接 |
| 可靠性 | 提供可靠传输 | 不保证可靠 |
| 顺序 | 保证按序交付 | 不保证顺序 |
| 重传 | 有 | 无 |
| 流量/拥塞控制 | 有 | 通常无 |
| 开销 | 较大 | 较小 |
| 典型场景 | Web、文件传输、数据库连接 | 视频、直播、DNS、实时游戏 |

面试中可以这样补一句：TCP 适合可靠性优先的场景，UDP 适合实时性或自定义可靠性机制的场景。

## 4. 面试回答模板

TCP 的可靠性不是靠单一机制实现的，而是一整套机制配合完成的。

首先，TCP 会给字节流中的数据编号，也就是序列号，接收方收到后会返回确认应答。这样发送方知道哪些数据已经成功到达，接收方也可以根据序列号做排序和去重。

其次，如果发送方在一定时间内没有收到确认，就会进行超时重传；如果连续收到重复 ACK，也可能触发快速重传。TCP 还会用校验和检查数据是否损坏。

然后，为了提高效率，TCP 使用滑动窗口，不需要每发一个包就停下来等确认。同时它还通过流量控制避免发送太快导致接收方处理不过来，通过拥塞控制避免网络已经很堵时继续大量发送。

所以总结起来，TCP 主要通过序列号、确认应答、重传机制、校验和、滑动窗口、流量控制和拥塞控制来保证可靠、有序地传输数据。

## 6. 常见误区

- **误区 1：只回答三次握手。** 三次握手是建立连接，不是全部可靠性机制。
- **误区 2：把流量控制和拥塞控制混为一谈。** 流量控制看接收方能力，拥塞控制看网络状况。
- **误区 3：认为 TCP 一定不会丢数据。** TCP 能检测和重传，但如果连接断开、应用异常或系统崩溃，仍然需要应用层处理异常。
- **误区 4：忽略有序性。** TCP 不只是保证“到达”，还保证应用层读到的数据顺序正确。

## 7. 记忆口诀

**编号确认会重传，校验排序控两端。**

解释：

- 编号：序列号
- 确认：ACK
- 重传：超时重传、快速重传
- 校验：校验和
- 排序：按序交付
- 控两端：流量控制管接收方，拥塞控制管网络

TCP 通过 **序列号、确认应答、重传机制、校验和、滑动窗口、流量控制和拥塞控制** 来保证可靠性。序列号保证顺序和去重，ACK 告诉发送方哪些数据到了，丢包就重传，校验和检查损坏，窗口机制提升效率，流量控制保护接收方，拥塞控制保护网络。
