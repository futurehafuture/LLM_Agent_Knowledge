# Redis 高可用

## 1. 面试官到底想考什么

高可用相关的问题，面试官想确认：

- 你是否了解单机 Redis 的局限性（单点故障、容量有限）。
- 你能否区分主从复制、哨兵、Cluster 三种方案，以及它们分别解决什么问题。
- 你是否知道数据分片和集群选举的基本原理。

## 2. 三种方案对比

| 方案 | 解决的问题 | 是否高可用 | 是否支持扩容 |
|------|-----------|-----------|------------|
| 主从复制 | 读写分离，数据备份 | 不自动（需手动切换） | 有限（读扩展） |
| 哨兵（Sentinel） | 自动故障转移 | 是 | 有限（读扩展） |
| Redis Cluster | 数据分片 + 高可用 | 是 | 是（水平扩展） |

## 3. 主从复制

### 3.1 基本概念

主从复制（Replication）是 Redis 高可用的基础。

```text
主节点（Master）：处理读写请求，数据变化同步给从节点
从节点（Slave/Replica）：只读，从主节点同步数据
```

**作用：**

1. **数据备份**：从节点是主节点的完整副本。
2. **读写分离**：写请求走主节点，读请求分散到从节点，提高读吞吐量。
3. **故障恢复基础**：主节点挂了，从节点可以提升为新主节点（哨兵/Cluster 做这件事）。

### 3.2 配置方式

```bash
# 从节点 redis.conf 中配置
replicaof 192.168.1.10 6379    # 主节点的 IP 和端口
# 或在运行时执行
REPLICAOF 192.168.1.10 6379
```

### 3.3 全量复制流程（首次同步）

```text
从节点向主节点发送 PSYNC 命令
主节点执行 BGSAVE，生成 RDB 快照
主节点把 RDB 文件发送给从节点（同时缓冲期间新来的写命令到 repl_backlog）
从节点加载 RDB 文件
主节点把缓冲的增量命令发给从节点
从节点执行增量命令，与主节点保持同步
```

### 3.4 增量复制（断线重连）

从节点断线后重连，不需要重新全量同步。主节点维护一个环形 **replication backlog buffer**（默认 1MB），从节点重连时传入上次同步的偏移量（offset），主节点从 backlog 中补发断线期间的命令。

```text
如果断线时间过长，offset 对应的数据已经被 backlog 覆盖 → 触发全量复制
如果断线时间短，backlog 中还有数据 → 只补发增量
```

### 3.5 主从复制的局限

- **主节点挂了，需要手动切换**：DBA 手动执行 `REPLICAOF NO ONE` 将从节点提升为主节点，应用层也要更新连接地址。
- **写操作无法扩展**：多台服务器都只读主节点，写还是只有一台机器处理。
- **全量复制时网络和内存压力大**。

## 4. 哨兵（Sentinel）

### 4.1 哨兵是什么

哨兵是 Redis 提供的**高可用方案**，在主从复制的基础上，增加了自动故障检测和自动故障转移能力。

```text
哨兵节点（Sentinel）：特殊的 Redis 进程，负责监控、选举和通知
Redis 节点：正常的 Redis 数据节点（主 + 从）
```

通常部署奇数个哨兵（3 个最常见），避免脑裂（split-brain）问题。

### 4.2 哨兵的三个核心功能

**1. 监控（Monitoring）**

每个哨兵每隔 1 秒向主节点、从节点、其他哨兵发送 PING 命令，检查是否存活。

```text
主观下线（SDOWN）：某个哨兵认为主节点不可达
客观下线（ODOWN）：多数哨兵（quorum）都认为主节点不可达，才真正判定下线
```

**2. 通知（Notification）**

哨兵可以通过发布订阅通知客户端，主节点地址发生了变化。

**3. 自动故障转移（Automatic Failover）**

主节点被判定客观下线后：

```text
1. 哨兵们通过 Raft 协议选举出一个"领头哨兵"（Leader Sentinel）
2. 领头哨兵从从节点中选出最合适的一个，提升为新主节点
3. 其他从节点修改复制目标，指向新主节点
4. 原主节点如果恢复，以从节点身份加入集群
5. 哨兵通知客户端新的主节点地址
```

**如何选择新主节点？** 按以下优先级：

```text
1. 过滤与主节点断开超过 down-after-milliseconds * 10 的从节点（数据太旧）
2. 选择 slave-priority 最高的（配置项，默认 100）
3. 选择复制偏移量最大的（数据最新的）
4. 选择 runid 字典序最小的（随机打破平局）
```

### 4.3 哨兵配置示例

```conf
# sentinel.conf
sentinel monitor mymaster 192.168.1.10 6379 2   # 监控主节点，quorum=2（需要2个哨兵同意才客观下线）
sentinel down-after-milliseconds mymaster 5000   # 5秒无响应判定主观下线
sentinel failover-timeout mymaster 60000         # 故障转移超时 60 秒
sentinel parallel-syncs mymaster 1              # 故障转移时，1个从节点同时同步新主节点
```

### 4.4 哨兵的局限

- **不支持水平扩容**：哨兵模式下，仍然只有一个主节点处理写，数据量和写 QPS 还是受单机限制。
- **主从切换过程有短暂不可用**（通常秒级）。
- 客户端需要通过哨兵获取主节点地址，有一定配置复杂度。

## 5. Redis Cluster（集群）

### 5.1 集群解决什么问题

哨兵模式仍然是单主节点，无法突破单台机器的内存上限。

Redis Cluster 通过**数据分片**，把数据分散到多个主节点，实现：

- **水平扩容**：增加节点就能增加容量和吞吐。
- **高可用**：每个主节点有对应的从节点，主节点挂了自动切换到从节点。

### 5.2 数据分片：哈希槽（Hash Slot）

Redis Cluster 把键空间划分为 **16384 个哈希槽**（Hash Slot），每个主节点负责一部分槽。

**计算槽的公式：**

```text
slot = CRC16(key) % 16384
```

例如，3 个主节点，每个节点分配约 5461 个槽：

```text
节点 A：0 ~ 5460
节点 B：5461 ~ 10922
节点 C：10923 ~ 16383
```

访问一个 key 时：

```text
客户端计算 key 的哈希槽
根据槽找到负责的节点
向那个节点发送命令
```

**为什么是 16384 个槽，不是更多？**

- 16384 个槽能支持约 1000 个节点（每个节点负责约 16 个槽）。
- 心跳包需要传输槽位信息，16384 个槽的位图只需 2KB，不会太大。

### 5.3 MOVED 重定向

如果客户端访问了错误的节点，节点会返回 MOVED 错误，告知正确节点地址：

```text
客户端: GET user:1001
节点 A: MOVED 10293 192.168.1.12:6379  ← 这个 key 在节点 C（槽 10293）
客户端: 重新向节点 C 发送 GET user:1001
```

智能客户端（如 redis-py、Jedis）会缓存槽位映射，减少重定向次数。

### 5.4 ASK 重定向

集群扩容/缩容时，槽在迁移中（部分数据已迁移），会返回 ASK 错误：

```text
ASK 10293 192.168.1.12:6379    ← 先发 ASKING 命令，再向目标节点请求
```

ASK 和 MOVED 的区别：

- MOVED：槽已经永久迁移，更新缓存。
- ASK：槽正在迁移中，这次请求去目标节点，但不更新缓存。

### 5.5 集群节点发现与心跳（Gossip 协议）

集群节点之间用 **Gossip 协议**相互通信，每个节点定期向其他节点发送心跳包（PING），包含：

- 自己负责的槽位信息。
- 其他节点的状态（从哪里听说的）。

通过这种"八卦传播"，每个节点最终都能了解整个集群的状态。

### 5.6 集群选举

某个主节点挂了，其对应的从节点会发起选举：

```text
从节点检测到主节点下线
向其他主节点发送拉票请求（FAILOVER_AUTH_REQUEST）
其他主节点投票（每个主节点在一个选举周期内只投一票）
获得 >N/2 + 1 票的从节点成为新主节点
```

这是一个基于 Raft 的选举机制。

**集群要求至少 3 个主节点**，否则无法完成多数票选举。

### 5.7 集群的局限

- **不支持跨节点多键操作**：MGET、MSET 如果多个 key 不在同一节点，会失败。
- **不支持跨节点事务**：MULTI/EXEC 只能操作同一节点的键。
- **客户端需要支持集群协议**。

**解决跨节点多键问题——哈希标签（Hash Tag）：**

```bash
# 用 {} 包裹哈希计算部分，让相关 key 落在同一个槽
SET {user:1001}:name "Alice"
SET {user:1001}:age 25
# CRC16 只计算 {} 内的部分，所以两个 key 落在同一个槽
```

### 5.8 集群部署结构

生产环境标准配置：3 主 3 从（最小集群）

```text
Master A (槽 0-5460)    ← → Slave A
Master B (槽 5461-10922) ← → Slave B
Master C (槽 10923-16383) ← → Slave C
```

## 6. 三种方案选型

| 场景 | 推荐方案 | 说明 |
|------|---------|------|
| 数据量小（< 数十 GB），读多写少 | 主从 + 哨兵 | 简单，易运维，足够满足需求 |
| 数据量大，需要扩容 | Redis Cluster | 数据分片，水平扩展 |
| 开发/测试环境 | 单机 | 简单省事 |
| 高频写 + 海量数据 | Redis Cluster | 多主节点分担写压力 |

## 7. 面试回答模板

Redis 高可用有三个层次：

第一层是**主从复制**，主节点处理读写，从节点同步数据，用于读写分离和数据备份，但主节点故障需要手动切换。

第二层是**哨兵（Sentinel）**，在主从复制基础上增加了自动监控和故障转移，主节点挂了后哨兵自动选举新主节点，实现高可用，但写能力仍受单主节点限制。

第三层是 **Redis Cluster**，通过哈希槽把数据分散到多个主节点，支持水平扩容，同时每个主节点有从节点做高可用，主节点挂了通过选举自动切换。Cluster 使用 16384 个哈希槽，用 CRC16(key) % 16384 决定每个 key 属于哪个节点。

## 我应该背下来的最短答案

主从复制提供数据备份和读写分离，但主节点故障需手动切换；哨兵在此基础上实现自动故障转移；Redis Cluster 用哈希槽（16384个）将数据分片到多主节点，实现水平扩容和高可用。生产环境中小数据量用主从+哨兵，大数据量用 Cluster。
