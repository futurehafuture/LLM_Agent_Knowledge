# Redis 持久化

## 1. 面试官到底想考什么

问 Redis 持久化，面试官想确认你理解：

- Redis 是内存数据库，为什么数据不会随重启丢失（答案：持久化）。
- RDB 和 AOF 的原理区别，以及各自的优缺点。
- 实际生产中应该选哪种方案，能说出理由。

## 2. 零基础版解释

Redis 把数据放在内存，但内存是掉电丢失的。如果 Redis 重启（服务器崩溃、重启机器），内存里的数据就没了。

**持久化**就是把内存里的数据定期保存到硬盘，让 Redis 重启后能从硬盘恢复数据。

Redis 提供两种主要持久化方式：

- **RDB（快照）**：定时给内存"拍照"，把快照存成文件。
- **AOF（追加日志）**：把每次写命令记录到日志文件，重启时重放日志恢复数据。

打个比方：

- RDB 像 Word 文档的"自动保存"，每隔一段时间保存整份文档。
- AOF 像记操作日志，把你的每一个输入都记下来，恢复时把操作重演一遍。

## 3. RDB（Redis Database）

### 3.1 工作原理

RDB 在指定时间点把内存中所有数据生成一个**快照文件**（默认叫 `dump.rdb`），存到磁盘。

**触发方式有两种：**

**主动触发：**

```bash
SAVE    # 阻塞主线程执行快照（期间 Redis 不响应其他命令）
BGSAVE  # 后台异步执行快照（推荐，主线程不阻塞）
```

**自动触发（配置文件 redis.conf）：**

```conf
# 格式：save <seconds> <changes>
# 在 seconds 秒内发生了 changes 次写操作，就自动触发 BGSAVE
save 900 1      # 900 秒内有 1 次写操作
save 300 10     # 300 秒内有 10 次写操作
save 60 10000   # 60 秒内有 10000 次写操作
```

满足任一条件就触发一次 BGSAVE。

### 3.2 BGSAVE 的 fork 机制

BGSAVE 是如何做到不阻塞主线程的？答案是 **fork（）系统调用 + Copy-On-Write（写时复制）**。

```text
主进程（父进程）
   ↓ 调用 fork()
子进程诞生
子进程执行快照写入磁盘
   ↓ 完成后退出
主进程继续处理命令
```

**写时复制（COW）的关键：**

fork 之后，父子进程**共享同一份内存**，只有当父进程（Redis 主线程）修改某块内存时，才会复制那块内存给子进程保留旧值，子进程继续用旧值写快照。

```text
fork 后：
父进程和子进程共用同一块内存（共享页）
↓
父进程收到写命令，要修改某个 key
↓
操作系统复制那一页内存给父进程（COW）
父进程修改自己的副本
子进程继续看到旧值，正常写快照
↓
快照完成后子进程退出
```

这样，快照期间 Redis 可以继续处理命令，数据一致性由 COW 保证。

**注意：** fork 时如果内存很大，fork 本身需要一点时间（因为需要复制页表），这期间主线程会有短暂阻塞。

### 3.3 RDB 文件格式

RDB 文件是**二进制格式**，紧凑高效，比 AOF 小得多，适合备份和快速恢复。

### 3.4 RDB 优缺点

**优点：**

- 文件紧凑，恢复速度快（直接加载快照，比 AOF 快很多）。
- 对 Redis 性能影响小（BGSAVE 在子进程中进行）。
- 适合全量备份（每天保存一个 dump.rdb）。

**缺点：**

- **数据丢失风险大**：如果在两次快照之间崩溃，最近的修改会丢失。例如每 5 分钟快照一次，崩溃前 4 分钟的数据就丢了。
- fork 时如果内存很大，可能有短暂停顿（毫秒到秒级，取决于内存大小）。

## 4. AOF（Append Only File）

### 4.1 工作原理

AOF 把 Redis 执行的每一条**写命令**追加到日志文件（默认叫 `appendonly.aof`）。

```bash
# 开启 AOF（redis.conf）
appendonly yes
appendfilename "appendonly.aof"
```

**恢复时**：Redis 重启后读取 AOF 文件，把里面的命令重新执行一遍，恢复到崩溃前的状态。

### 4.2 写回策略（fsync 策略）

AOF 记录命令的关键问题是：写命令写到 AOF 文件的时机。有三种策略：

```conf
appendfsync always    # 每次写命令立即 fsync 到磁盘（最安全，最慢）
appendfsync everysec  # 每秒 fsync 一次（默认，推荐，最多丢 1 秒数据）
appendfsync no        # 由操作系统决定何时 fsync（最快，最不安全）
```

| 策略 | 数据安全性 | 性能 | 说明 |
|------|-----------|------|------|
| always | 最高（不丢数据） | 最差 | 每次写都刷盘，磁盘 IO 压力大 |
| everysec | 高（最多丢 1 秒） | 好 | 推荐，平衡安全与性能 |
| no | 最低（可能丢较多） | 最好 | 依赖 OS，不可控 |

### 4.3 AOF 重写（Rewrite）

AOF 日志只追加不修改，随着运行时间增长，文件会越来越大。例如：

```bash
SET counter 1
INCR counter      # counter = 2
INCR counter      # counter = 3
...
INCR counter      # counter = 1000
```

这 1000 条命令最终可以压缩为一条 `SET counter 1000`。

**AOF 重写**就是这个压缩过程，生成一个更紧凑的 AOF 文件。

**触发方式：**

```bash
BGREWRITEAOF     # 手动触发后台重写
```

```conf
# 自动触发（redis.conf）
auto-aof-rewrite-percentage 100    # 当 AOF 文件比上次重写后大 100% 时
auto-aof-rewrite-min-size 64mb     # 且 AOF 文件超过 64MB 时
```

**重写过程（类似 BGSAVE）：**

```text
主进程 fork 子进程
子进程读取当前内存状态，生成精简版 AOF（新文件）
主进程继续处理命令，新命令写入旧 AOF 和重写缓冲区
子进程写完后，主进程把重写缓冲区的命令追加到新文件
原子替换旧 AOF 文件
```

### 4.4 AOF 优缺点

**优点：**

- **数据丢失少**：everysec 策略最多丢 1 秒数据，always 策略几乎不丢。
- 文件是文本命令格式，可读性强，便于排查问题或手动修复。
- 支持重写，防止文件无限增大。

**缺点：**

- **文件比 RDB 大**：AOF 是命令日志，比快照大很多。
- **恢复速度慢**：重启需要重放所有命令，数据量大时恢复时间长。
- 某些命令重放可能有细微问题（如 EXPIRE 时间的精确性）。

## 5. RDB vs AOF 对比

| 对比项 | RDB | AOF |
|--------|-----|-----|
| 数据完整性 | 差（可能丢失最后一次快照后的数据） | 好（最多丢 1 秒，可配置不丢） |
| 文件大小 | 小（二进制快照） | 大（命令日志，重写后缩小） |
| 恢复速度 | 快 | 慢（需重放命令） |
| 对性能影响 | fork 时有短暂停顿 | 每秒 fsync 有轻微 IO 压力 |
| 适合场景 | 备份，允许分钟级数据丢失 | 对数据丢失敏感的场景 |

**如果两个都开启，Redis 重启时优先使用 AOF 文件恢复**，因为 AOF 的数据更完整。

## 6. 混合持久化（Redis 4.0+）

混合持久化结合了 RDB 和 AOF 的优点：

```conf
aof-use-rdb-preamble yes    # 开启混合持久化
```

**工作方式：**

AOF 重写时，文件内容变为：

```text
[RDB 格式的快照数据] + [快照之后的 AOF 格式增量命令]
```

**恢复时：**

1. 先快速加载 RDB 部分（速度快）。
2. 再重放 AOF 增量部分（数量少，快速）。

```text
优点：恢复速度接近 RDB，数据完整性接近 AOF
缺点：AOF 文件不再是纯文本，可读性降低
```

生产环境推荐使用混合持久化。

## 7. 实际生产配置建议

### 7.1 对数据安全要求高（不允许丢失超过 1 秒）

```conf
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes    # 混合持久化
save ""                      # 禁用 RDB 自动快照（可选，减少性能抖动）
```

### 7.2 允许丢失分钟级数据（如缓存场景）

```conf
appendonly no    # 只用 RDB
save 300 10
save 60 10000
```

### 7.3 纯缓存（允许丢失全部数据）

```conf
appendonly no
save ""          # 禁用所有持久化
```

适合：Redis 只做缓存，数据丢失后从 MySQL 重新加载。

## 8. 面试中的延伸问题

### Q：Redis 重启后如何恢复数据？

```text
1. 如果同时开启了 RDB 和 AOF，优先使用 AOF 恢复。
2. 如果只有 RDB，加载 dump.rdb 文件。
3. 如果只有 AOF，重放 appendonly.aof。
4. 如果都没有，Redis 启动后数据为空。
```

### Q：持久化对性能有多大影响？

- RDB：主要影响来自 fork 时间。内存 10GB 以下通常 fork 在 100ms 以内。
- AOF everysec：每秒一次 fsync，对磁盘 IO 有一定压力，通常在可接受范围内。
- AOF always：每次写命令都 fsync，写入 QPS 受限于磁盘写入速度，通常 < 1万 QPS。

## 9. 面试回答模板

Redis 持久化有两种主要方式：RDB 和 AOF。

RDB 是快照持久化，定时把内存数据生成二进制快照文件，通过 fork 子进程异步执行，不阻塞主线程。RDB 文件小、恢复快，但两次快照之间崩溃会丢数据。

AOF 是追加写日志，把每次写命令追加到日志文件，通过 everysec 策略最多丢失 1 秒数据，安全性更高，但文件更大，恢复更慢。

Redis 4.0+ 支持混合持久化，AOF 重写时文件前半段是 RDB 快照，后半段是增量 AOF 命令，兼顾恢复速度和数据安全性。生产环境通常推荐混合持久化 + everysec 策略。

## 我应该背下来的最短答案

RDB 是定时快照，文件小恢复快，但可能丢失最近一次快照后的数据；AOF 是命令日志，数据更安全（最多丢 1 秒），但文件大恢复慢。两者同时开启时优先用 AOF 恢复。Redis 4.0+ 混合持久化兼顾两者优点，生产环境推荐。
