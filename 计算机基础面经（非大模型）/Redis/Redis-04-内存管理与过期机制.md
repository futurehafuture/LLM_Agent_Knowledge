# Redis 内存管理与过期机制

## 1. 面试官到底想考什么

这类题考察的是：

- 你是否理解 Redis 如何管理内存有限的资源。
- 你能否说清楚"过期键是什么时候删除的"——很多人以为过期就立刻删，其实不是。
- 当 Redis 内存打满时，你知道它会怎么处理，有哪些淘汰策略。

## 2. 零基础版解释

Redis 数据放在内存里，内存是有限的。如何管理内存，有两个核心问题：

1. **键过期了（超过 TTL）什么时候删除？**
2. **内存用满了怎么办？**

对于第一个问题，Redis 不会"立刻删除过期键"，而是用**惰性删除 + 定期删除**的组合策略。

对于第二个问题，Redis 用**内存淘汰策略**，在内存不足时按照一定规则删除一些数据。

## 3. 过期键删除策略

### 3.1 为什么需要策略

当你执行 `SET key value EX 3600`，Redis 会记录这个键的过期时间。3600 秒后这个键"逻辑上"已经过期。

但 Redis 是否立刻删除它？**不一定**。因为：

- 如果有 10 万个键同时过期，立刻删除会占用大量 CPU。
- Redis 需要平衡 CPU 使用率和内存使用率。

Redis 使用**两种策略结合**来处理过期键。

### 3.2 惰性删除（Lazy Deletion）

**原理：** 不主动删除，等到用户访问这个键时，才检查是否过期，如果过期了就删除并返回空。

```text
用户执行 GET key
   ↓
Redis 检查 key 是否过期
   ↓ 未过期 → 返回值
   ↓ 已过期 → 删除 key → 返回 nil
```

**优点：** 对 CPU 友好，不需要主动扫描，只有访问时才检查。

**缺点：** 如果过期键一直没被访问，它就会一直占着内存，造成内存泄漏。

### 3.3 定期删除（Periodic Deletion）

**原理：** Redis 每隔一段时间，随机抽取一批设置了过期时间的键，检查并删除其中已过期的。

```text
每隔 100ms（默认），执行一次扫描：
1. 随机从过期字典中抽取 20 个键
2. 删除其中已过期的键
3. 如果删除比例超过 25%，继续扫描（最多执行 20ms）
4. 否则结束本次扫描
```

这样设计的好处是：每次扫描的时间有上限（20ms），不会占用太多 CPU。

**配置项：**

```conf
hz 10    # 每秒执行 10 次后台任务（默认），提高这个值会更积极地删除过期键
         # 但会增加 CPU 使用率
```

### 3.4 两种策略的配合

```text
惰性删除：保证被访问的过期键立刻清除（精准，低CPU）
定期删除：定期清理未被访问的过期键（兜底，避免内存泄漏）
```

两者组合让 Redis 在 CPU 和内存之间取得平衡。

## 4. 过期时间相关命令

```bash
# 设置过期时间
EXPIRE key 3600          # 设置 3600 秒后过期
EXPIREAT key 1718380800  # 设置在指定 Unix 时间戳过期
PEXPIRE key 3000         # 毫秒级
PEXPIREAT key timestamp  # 毫秒级时间戳

# 查询
TTL key                  # 查询剩余时间（秒），-1 表示永不过期，-2 表示键不存在
PTTL key                 # 毫秒级

# 移除过期时间
PERSIST key              # 移除过期时间，变为永久键
```

**注意事项：**

```bash
# 对一个有过期时间的 key 执行 SET，会清除过期时间！
SET key "new_value"     # 过期时间被清除
# 如果想保留过期时间，应该 GETSET 或者重新 EXPIRE
```

## 5. 内存淘汰策略（maxmemory-policy）

### 5.1 为什么需要淘汰策略

即使有过期键删除，如果：

- 写入速度超过过期/删除速度
- 没有设置过期时间的键大量积累
- 内存配置较小

就可能出现内存用满的情况。这时 Redis 需要根据策略淘汰一些数据。

### 5.2 maxmemory 配置

```conf
maxmemory 4gb    # 最大内存 4GB，超过后触发淘汰策略
```

如果不配置，Redis 会一直使用内存直到耗尽（生产环境必须配置）。

### 5.3 八种淘汰策略

Redis 提供 8 种 `maxmemory-policy`：

**不淘汰（直接报错）：**

| 策略 | 说明 |
|------|------|
| `noeviction` | 内存满了，写操作直接报错（默认值） |

**只从设置了过期时间的键里淘汰（volatile 系列）：**

| 策略 | 说明 |
|------|------|
| `volatile-lru` | 从有过期时间的键中，淘汰最近最少使用的（LRU） |
| `volatile-lfu` | 从有过期时间的键中，淘汰使用频率最低的（LFU，Redis 4.0+） |
| `volatile-random` | 从有过期时间的键中，随机淘汰 |
| `volatile-ttl` | 从有过期时间的键中，淘汰剩余存活时间最短的 |

**从所有键里淘汰（allkeys 系列）：**

| 策略 | 说明 |
|------|------|
| `allkeys-lru` | 从所有键中，淘汰最近最少使用的（最常用） |
| `allkeys-lfu` | 从所有键中，淘汰使用频率最低的（Redis 4.0+） |
| `allkeys-random` | 从所有键中，随机淘汰 |

### 5.4 LRU vs LFU

**LRU（Least Recently Used，最近最少使用）：**

淘汰**最久没被访问**的键。

```text
时间轴：key1(10分钟前) key2(5分钟前) key3(1分钟前)
LRU 淘汰：key1（最久没访问）
```

**问题：** 如果某个 key 访问频率很低，但刚好最近被访问了一次，就不会被淘汰，反而会占用内存。

**LFU（Least Frequently Used，最少频率使用）：**

淘汰**历史访问频率最低**的键。

```text
key1：访问 1000 次（但最近没访问）
key2：访问 2 次（最近刚访问了）
LFU 淘汰：key2（访问频率低）
```

**两者比较：**

| 对比 | LRU | LFU |
|------|-----|-----|
| 考虑因素 | 最近是否访问 | 历史访问频率 |
| 抗"偶发冷数据"能力 | 弱（刚被访问就不淘汰） | 强 |
| 实现复杂度 | 简单 | 稍复杂 |
| 适用场景 | 通用 | 冷热数据分布明显的场景 |

**Redis 的 LRU 是近似 LRU：**

严格 LRU 需要一个链表跟踪所有键的访问顺序，内存开销大。Redis 用**近似 LRU**：随机采样 N 个键（默认 5 个），淘汰其中最久未访问的一个。`maxmemory-samples` 配置采样数量，越大越精准但 CPU 消耗更多。

### 5.5 如何选择淘汰策略

**场景对应推荐：**

| 场景 | 推荐策略 | 理由 |
|------|---------|------|
| 纯缓存，所有键都会过期 | `allkeys-lru` 或 `allkeys-lfu` | 充分利用内存，热点数据留下来 |
| 部分键永不过期（配置、字典等） | `volatile-lru` | 只淘汰可以重建的缓存键，保留永久键 |
| 对一致性要求高（数据库兼用） | `noeviction` | 宁可报错也不丢数据 |
| 热点数据分布明显 | `allkeys-lfu` | 高频热点保留，低频冷数据淘汰 |

## 6. 内存优化实践

### 6.1 合理设置过期时间

永远不要让缓存键无过期时间（除非是元数据配置等确实需要持久的数据）：

```python
# 错误：不设置过期时间，内存会无限增长
redis.set(f"user:info:{user_id}", user_data)

# 正确：根据业务需要设置合理过期时间
redis.setex(f"user:info:{user_id}", 3600, user_data)
```

### 6.2 选择合适的数据类型

小对象用 Hash 比用多个 String 更节省内存：

```bash
# 效率低：多个 String 键，每个键都有元数据开销
SET user:1001:name "Alice"
SET user:1001:age 25
SET user:1001:city "Shanghai"

# 效率高：一个 Hash，共享键的元数据
HSET user:1001 name "Alice" age 25 city "Shanghai"
```

### 6.3 监控内存使用

```bash
INFO memory                      # 查看内存详情
DEBUG OBJECT key                 # 查看单个键的内存占用
MEMORY USAGE key                 # 查看单个键的内存（Redis 4.0+）
MEMORY STATS                     # 内存统计摘要
```

关注字段：

- `used_memory`：Redis 实际使用的内存
- `maxmemory`：配置的最大内存
- `mem_fragmentation_ratio`：内存碎片率（正常 1.0~1.5，过高说明有碎片）

## 7. 面试回答模板

Redis 的过期键删除使用两种策略结合：**惰性删除**（访问时检查是否过期）和**定期删除**（每 100ms 随机抽取一批过期键删除）。两者结合在 CPU 和内存之间取得平衡。

当内存超过 maxmemory 限制时，Redis 根据配置的 maxmemory-policy 淘汰键。常用策略有：noeviction（写入报错）、allkeys-lru（所有键中淘汰最近最少使用）、volatile-lru（只从有过期时间的键中淘汰）等。纯缓存场景一般用 allkeys-lru 或 allkeys-lfu，混合存储场景用 volatile-lru 保护永久数据。

## 我应该背下来的最短答案

过期键删除：惰性删除（访问时检查）+ 定期删除（每隔 100ms 随机扫描）。内存淘汰：maxmemory 配置上限，超出后按 maxmemory-policy 策略淘汰，常用 allkeys-lru（纯缓存）或 volatile-lru（混合存储）。
