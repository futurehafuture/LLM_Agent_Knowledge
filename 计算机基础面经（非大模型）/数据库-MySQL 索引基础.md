# MySQL 索引基础

## 1. 面试官到底想考什么

面试官问索引，通常想看你是否理解数据库查询为什么能变快，以及变快背后付出了什么代价。

很多初学者会说“索引可以加快查询”，这只答到一半。真正更完整的回答应该包含：**索引是什么、为什么快、有哪些类型、什么时候会失效、有什么缺点**。

## 2. 零基础版解释

索引就像书的目录。

如果没有目录，你想找“TCP”这一章，只能从第一页翻到最后一页。数据库没有索引时，也可能需要一行一行扫描，这叫全表扫描。

如果有目录，你可以先看目录，知道“TCP”在哪一页，然后直接翻过去。数据库有索引时，可以先通过索引定位到数据位置，再去读取数据。

但是目录也有成本：书越厚，目录也越长；每次新增章节，目录也要更新。数据库索引也是一样，能加快查询，但会占空间，并且插入、删除、更新时需要维护索引。

## 3. 正式概念

**索引** 是数据库为了加速查询而维护的一种数据结构。它保存了字段值和数据位置之间的映射，让数据库可以更快定位目标记录。

MySQL 中常见索引概念：

- **主键索引**：主键字段上的索引，唯一且非空。
- **唯一索引**：索引列的值不能重复。
- **普通索引**：只加速查询，不限制唯一性。
- **联合索引**：多个字段组合建立的索引。
- **聚簇索引**：InnoDB 中，表数据和主键索引放在一起，叶子节点保存整行数据。
- **二级索引**：非主键索引，叶子节点通常保存主键值，需要时再通过主键查整行数据，这个过程叫回表。

索引的缺点：

- **占用额外存储空间**。
- **降低写入速度**，因为 insert、update、delete 时要维护索引。
- **索引过多会增加优化器选择成本**。
- **不合适的索引可能没有效果**，甚至拖慢查询。
- **范围查询、函数计算、隐式类型转换等情况可能导致索引失效或效果变差**。

### 3.1 没有索引时数据库怎么查

没有索引时，数据库常见做法是**全表扫描**。

例如有一张用户表：

```sql
SELECT * FROM user WHERE phone = '13800000000';
```

如果 `phone` 没有索引，数据库可能从第一行开始，一行一行比较 `phone` 是否等于目标值。表只有 100 行时无所谓，表有 1000 万行时就会很慢。

索引的价值就是：把“一行一行找”变成“按数据结构快速定位”。

### 3.2 InnoDB 的主键索引和二级索引

InnoDB 表通常按照主键组织数据。主键索引也叫**聚簇索引**。

聚簇索引特点：

- 叶子节点保存整行数据。
- 一张 InnoDB 表通常只有一个聚簇索引。
- 如果有主键，就用主键作为聚簇索引。
- 如果没有主键，InnoDB 会选择唯一非空索引或内部隐藏字段。

二级索引特点：

- 也叫辅助索引、普通索引。
- 叶子节点通常保存索引列值和主键值。
- 如果查询需要整行数据，需要拿主键再去主键索引查一次。

这个再次查询主键索引的过程叫**回表**。

例子：

```sql
CREATE TABLE user (
  id BIGINT PRIMARY KEY,
  name VARCHAR(50),
  phone VARCHAR(20),
  age INT,
  INDEX idx_phone(phone)
);
```

查询：

```sql
SELECT * FROM user WHERE phone = '13800000000';
```

执行过程大致是：

1. 在 `idx_phone` 这棵二级索引上找到对应 phone。
2. 拿到这一行的主键 id。
3. 再到主键索引上根据 id 找到整行数据。

### 3.3 覆盖索引

如果查询需要的字段都在索引里，就不需要回表，这叫**覆盖索引**。

例如：

```sql
SELECT id, phone FROM user WHERE phone = '13800000000';
```

如果 `idx_phone(phone)` 的叶子节点里有 `phone` 和主键 `id`，那么查询 `id, phone` 可以直接从二级索引得到结果，不需要回表。

覆盖索引的好处：

- 减少一次主键索引查询。
- 降低磁盘 IO。
- 对高频查询很有价值。

### 3.4 联合索引和最左前缀原则

联合索引是多个字段一起建立的索引。

```sql
CREATE INDEX idx_name_age_city ON user(name, age, city);
```

这个索引的排序方式可以理解为：

```text
先按 name 排序
name 相同再按 age 排序
age 相同再按 city 排序
```

所以它通常可以支持：

```sql
WHERE name = 'Alice'
WHERE name = 'Alice' AND age = 20
WHERE name = 'Alice' AND age = 20 AND city = 'Shanghai'
```

但不适合直接跳过最左列：

```sql
WHERE age = 20
WHERE city = 'Shanghai'
```

这就是**最左前缀原则**。联合索引要从最左边字段开始连续使用。

### 3.5 索引失效的常见情况

索引不一定建了就会用。常见失效或效果变差的情况：

| 情况 | 示例 | 原因 |
|---|---|---|
| 对索引列使用函数 | `WHERE DATE(create_time) = '2026-06-02'` | 索引保存的是原值，不是函数结果 |
| 隐式类型转换 | 字符串列用数字查 | 类型转换可能破坏索引使用 |
| 左模糊匹配 | `LIKE '%abc'` | 无法从索引有序开头定位 |
| 联合索引跳过最左列 | 索引 `(a,b)` 只查 `b` | 不符合排序前缀 |
| 使用不等于 | `!=`、`NOT IN` | 选择性可能差 |
| OR 条件不合理 | 一个条件有索引，一个没有 | 可能导致全表扫描 |

更好的写法例子：

```sql
-- 不推荐
WHERE DATE(create_time) = '2026-06-02'

-- 推荐
WHERE create_time >= '2026-06-02 00:00:00'
  AND create_time <  '2026-06-03 00:00:00'
```

### 3.6 字段区分度

并不是所有字段都适合建索引。

**区分度**指的是字段值的分散程度。手机号、身份证号、用户 id 区分度很高；性别、是否删除、状态码区分度通常较低。

高区分度字段更适合索引，因为它能快速缩小结果范围。

低区分度字段不一定没用，但要结合场景。例如 `status` 单独建索引可能效果一般，但 `(status, create_time)` 这样的联合索引可能适合查某状态下最近的数据。

### 3.7 EXPLAIN 是什么

`EXPLAIN` 可以查看 SQL 的执行计划。

```sql
EXPLAIN SELECT * FROM user WHERE phone = '13800000000';
```

常关注：

- `type`：访问类型，是否全表扫描。
- `key`：实际使用了哪个索引。
- `rows`：预计扫描多少行。
- `Extra`：是否 Using index、Using where、Using filesort 等。

面试时不一定要背所有字段，但要知道：分析 SQL 慢不慢，不能只靠猜，要看执行计划。

## 4. 面试回答模板

MySQL 索引可以理解为数据库为了加速查询而维护的数据结构，类似书的目录。没有索引时，数据库可能要做全表扫描；有索引时，可以通过索引更快定位到符合条件的数据。

在 InnoDB 里，常见索引底层是 B+ 树。主键索引是聚簇索引，叶子节点保存整行数据；普通索引或者二级索引的叶子节点通常保存主键值，如果查询字段不在索引里，就需要再通过主键查一次，这叫回表。

索引的好处是提高查询效率，尤其是 where、join、order by、group by 等场景。但索引不是越多越好，因为它会占用额外空间，而且写入、删除、更新时都要维护索引，所以会降低写入性能。索引设计要结合查询条件、字段区分度和业务访问模式。

## 6. 常见误区

- **误区 1：认为索引只影响查询，不影响写入。** 写入时也要更新索引结构。
- **误区 2：认为每个字段都应该建索引。** 低区分度字段、很少查询的字段不一定适合建索引。
- **误区 3：不知道回表。** 二级索引查到主键后，可能还要再查主键索引。
- **误区 4：只会说 B+ 树，不会说为什么。** 要能联系磁盘 IO 和范围查询。

## 7. 记忆口诀

**索引像目录，查询快；写入要维护，空间也得买。**

## 我应该背下来的最短答案

MySQL 索引是为了加速查询维护的数据结构，类似书的目录。它能减少全表扫描，提高 where、join、排序和分组的效率。但索引会占额外空间，并且 insert、update、delete 时要维护索引，所以会降低写入性能。InnoDB 常用 B+ 树索引，主键索引叶子节点保存整行数据，二级索引可能需要回表。
