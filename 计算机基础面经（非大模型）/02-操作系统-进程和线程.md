# 进程和线程

## 1. 面试官到底想考什么

“进程和线程有什么区别”是一道入口题，但它背后串着操作系统最核心的几条线：

- **抽象**：程序、进程、线程分别是什么。
- **资源**：地址空间、打开文件、信号、环境变量等资源归谁管。
- **调度**：CPU 实际切换和调度的执行实体是什么。
- **隔离**：为什么一个进程崩溃通常不直接拖垮另一个进程。
- **通信与同步**：进程怎么通信，线程怎么避免共享数据被并发写坏。
- **Linux 实现**：为什么 Linux 里常说线程是轻量级进程。

最常见的开场白是：

> 进程是资源分配的基本单位，线程是 CPU 调度的基本单位。

这句话可以说，但不能停在这里。比较完整的回答要能解释：为什么进程“重”、线程“轻”；线程到底共享什么、不共享什么；多线程一定更快吗；Linux 里的 `fork()`、`exec()`、`clone()` 和 pthread 有什么关系。

## 2. 零基础版解释

一个**程序**是磁盘上的静态文件，比如浏览器可执行文件。你双击运行它，操作系统把程序加载到内存，为它分配资源，形成一个**进程**。

进程像一个独立公司：

- 有自己的办公区：虚拟地址空间。
- 有自己的资产：打开的文件、网络连接、环境变量等。
- 有自己的管理档案：PID、状态、父子关系、权限等。

线程像公司里的员工。一个公司可以有多个员工同时干活：

- 一个线程负责网络请求。
- 一个线程负责渲染页面。
- 一个线程负责响应用户输入。

员工共享公司的办公区和资源，但每个员工有自己的工作现场：当前执行到哪条指令、寄存器值、调用栈。

所以可以先记：

```text
程序：静态代码和数据
进程：运行中的程序 + 独立资源容器
线程：进程内部的一条执行流
```

## 3. 程序、进程、线程的严格区分

### 3.1 程序是静态文件

程序是磁盘上的可执行文件，包含代码段、静态数据、依赖信息等。程序本身不会占用 CPU 执行，只有被加载运行后才会变成进程。

例如：

```text
/usr/bin/python3     -> 程序
python3 app.py       -> 一个 Python 进程
python3 worker.py    -> 另一个 Python 进程
```

同一个程序可以启动多个进程。它们代码来自同一份文件，但运行时拥有各自的地址空间和运行状态。

### 3.2 进程是资源容器

进程是操作系统对“正在运行的程序”的抽象。一个进程通常包含：

- 独立虚拟地址空间。
- 代码段、数据段、堆、栈映射。
- 打开的文件描述符。
- 当前工作目录、环境变量。
- 用户/组权限、信号处理信息。
- 至少一个线程。
- 内核中的进程控制块，例如 Linux 的 `task_struct` 相关信息。

经典操作系统课程通常把进程描述为：地址空间、寄存器状态和内核维护的持久状态共同组成的运行实例。换成人话就是：进程不仅有代码，还要有“我运行到哪里了”和“我拥有哪些系统资源”。

### 3.3 线程是执行流

线程是进程里的执行单位。多个线程共享同一个进程的大部分资源，但每个线程要有独立的执行现场。

一个线程至少需要：

- 程序计数器 PC：下一条要执行的指令。
- CPU 寄存器上下文：被切换出去时要保存。
- 栈：保存函数调用、返回地址、局部变量等。
- 线程 ID、调度优先级、信号屏蔽字、线程局部存储等线程属性。

线程存在的核心价值是：让同一个进程内部可以有多条执行路径，提高并发性，尤其是 I/O 等待较多的程序。

## 4. 一句话区别和完整对比表

### 4.1 一句话版本

进程是操作系统进行资源隔离和资源管理的基本单位，线程是进程内部可被调度执行的基本单位。进程之间地址空间通常相互独立；同一进程内的线程共享地址空间和大部分进程资源，但每个线程有自己的栈、寄存器和程序计数器。

### 4.2 对比表

| 维度 | 进程 | 线程 |
|---|---|---|
| 本质 | 运行中的程序实例，资源容器 | 进程内的一条执行流 |
| 资源 | 拥有独立地址空间、文件表等资源 | 共享所属进程的大部分资源 |
| 调度 | 现代系统中进程至少包含一个可调度线程 | 通常是 CPU 调度的直接对象 |
| 地址空间 | 进程之间默认相互隔离 | 同一进程内线程共享地址空间 |
| 创建开销 | 较大，需要建立进程资源和地址空间 | 较小，复用进程资源 |
| 切换开销 | 通常较大，可能涉及地址空间和 TLB | 通常较小，同进程线程不换地址空间 |
| 通信方式 | 需要 IPC | 共享内存即可通信，但要同步 |
| 崩溃影响 | 隔离性好，一个进程崩溃通常不直接影响另一个 | 一个线程崩溃可能导致整个进程异常 |
| 典型场景 | 服务隔离、多进程模型、浏览器多进程架构 | I/O 并发、后台任务、并行计算 |

## 5. 线程共享什么，不共享什么

POSIX pthreads 的描述很适合面试：同一进程内多个线程执行同一个程序，共享全局内存，比如数据段和堆，但每个线程有自己的栈。

### 5.1 通常共享的内容

同一进程内线程通常共享：

- 代码段。
- 数据段、全局变量、静态变量。
- 堆内存。
- 虚拟地址空间。
- 打开的文件描述符。
- 当前工作目录和根目录。
- 进程 ID、父进程 ID、进程组、会话等进程级属性。
- 信号处理方式。
- 用户 ID、组 ID。
- 资源限制等进程级配置。

这就是线程通信方便的原因：大家可以直接读写同一块内存。

### 5.2 通常独立的内容

每个线程通常独立拥有：

- 栈。
- 程序计数器 PC。
- CPU 寄存器上下文。
- 线程 ID。
- `errno`。
- 信号屏蔽字。
- 调度策略和优先级。
- 线程局部存储 TLS。

这就是线程能独立执行的原因：每个线程都知道自己执行到哪里、当前函数调用链是什么、被切换回来后如何继续运行。

### 5.3 高频追问：局部变量一定线程安全吗

常见回答“局部变量在线程栈上，所以线程安全”只对一部分情况成立。

更准确地说：

- 函数内部普通局部变量通常在线程自己的栈帧里，不被其他线程共享，因此一般没有数据竞争。
- 但如果把局部变量的地址传给其他线程，或者局部变量指向堆上的共享对象，就可能不安全。
- `static` 局部变量不在线程栈上，而是静态存储期对象，多线程访问仍然要同步。

例子：

```c
void f() {
    int x = 0;      // 通常是当前线程栈上的局部变量
    static int y;   // 进程内共享的静态变量，多线程访问要小心
}
```

面试里可以这样答：局部变量是否安全，要看变量本身和它指向的数据是否被多个线程共享。

## 6. 为什么进程切换通常比线程切换重

上下文切换的核心是：把当前执行实体的现场保存起来，再恢复另一个执行实体的现场。OSTEP 对上下文切换的解释很直观：保存当前寄存器状态，恢复即将运行的执行实体的寄存器状态。

### 6.1 同进程线程切换

同一进程内两个线程切换，通常需要切换：

- PC。
- 通用寄存器。
- 栈指针。
- 线程调度状态。

但它们共享同一个虚拟地址空间，所以通常不需要切换页表，也不需要换成另一套进程地址空间。

### 6.2 进程切换

不同进程之间切换，除寄存器、栈等 CPU 上下文外，还可能涉及：

- 切换虚拟地址空间。
- 切换页表基址。
- 影响 TLB 命中率。
- 切换与进程相关的内核状态。
- 缓存局部性变差。

所以进程切换一般更重。不过注意两个细节：

- “更重”不是说一定慢很多，现代系统做了大量优化。
- 如果两个线程竞争锁、频繁阻塞、造成缓存抖动，多线程也可能很慢。

### 6.3 面试不要说死

更稳的说法是：

```text
同一进程内线程切换通常更轻，因为不需要切换整个地址空间；
进程切换通常更重，因为涉及更强隔离和更多内核状态，可能影响页表、TLB 和缓存局部性。
```

## 7. 进程通信 IPC 和线程同步

### 7.1 进程之间为什么需要 IPC

进程之间地址空间独立，不能像线程那样直接访问彼此内存。因此要通过操作系统提供的 IPC 机制通信。

常见 IPC：

| 方式 | 特点 | 常见场景 |
|---|---|---|
| 管道 Pipe | 半双工、亲缘进程常用 | shell 管道、父子进程通信 |
| FIFO 命名管道 | 有路径名，无亲缘关系也可通信 | 简单本机进程通信 |
| 消息队列 | 按消息传递，内核维护队列 | 结构化消息通信 |
| 共享内存 | 多进程映射同一块内存 | 高性能、大数据量通信 |
| 信号 Signal | 异步通知，信息量少 | 终止、暂停、定时通知 |
| Socket | 可跨机器，也可本机 | 客户端/服务端、微服务通信 |
| mmap | 文件或匿名映射共享 | 共享文件、共享内存 |

### 7.2 哪种 IPC 最快

常见面试答案是共享内存最快，因为数据不需要在内核和用户态之间反复复制。

但要补一句：共享内存只解决“数据放在哪里”，不自动解决“谁先读、谁先写”。多个进程访问共享内存时仍然需要信号量、互斥锁、文件锁、futex 等同步机制。

### 7.3 线程之间为什么重点是同步

同一进程内线程共享内存，所以通信很方便，真正麻烦的是并发安全。

常见同步机制：

- 互斥锁 Mutex：保护临界区。
- 读写锁 RWLock：读多写少场景。
- 条件变量 Condition Variable：等待某个条件发生。
- 信号量 Semaphore：控制资源数量。
- 原子操作 Atomic：处理简单计数或状态更新。
- 屏障 Barrier：等待一组线程都到达某点。

### 7.4 竞态条件例子

两个线程同时执行：

```c
counter++;
```

它看起来是一行，但底层可能是：

```text
读取 counter
加 1
写回 counter
```

如果两个线程交错执行，可能两个都读到旧值，最后只加了一次。这就是竞态条件。解决办法是用锁或原子操作让这段更新变成安全的临界区。

## 8. Linux 里的进程和线程

### 8.1 Linux 的核心执行实体是 task

在 Linux 内核里，进程和线程都用任务来描述，关键结构常被称为 `task_struct`。这也是为什么很多资料说 Linux 线程是轻量级进程。

从实现角度看，Linux 不是先做一个完全不同的“线程对象”，而是用 `clone()` 系统调用创建任务，并通过不同 flag 决定新任务和父任务共享哪些资源。

### 8.2 `fork()`、`exec()`、`clone()` 的关系

可以这样理解：

- `fork()`：创建一个子进程，子进程拥有父进程地址空间的副本；现代系统通常用写时复制优化。
- `exec()`：用新程序替换当前进程的地址空间，常见于 shell 启动新程序。
- `clone()`：Linux 更底层、更灵活的创建接口，可以指定共享地址空间、文件描述符、信号处理等资源。

常见组合：

```text
fork() + exec()：创建新进程并运行新程序
clone(CLONE_VM 等)：创建共享地址空间的执行实体，也就是线程风格任务
```

### 8.3 `CLONE_VM` 和 `CLONE_THREAD`

在 Linux `clone(2)` 文档里，`CLONE_VM` 表示调用方和子任务运行在同一内存空间中；`CLONE_THREAD` 会把子任务放进同一个线程组。

所以 Linux 线程不是魔法，本质上是共享更多资源的 task：

```text
传统进程：资源共享少，隔离强
线程：共享地址空间、文件描述符等资源，切换和通信更轻
```

### 8.4 PID、TGID、TID 怎么区分

Linux 里容易混的概念：

- **PID**：在内核任务视角，每个 task 都有标识。
- **TID**：线程 ID，常指具体线程/任务的 ID。
- **TGID**：线程组 ID，通常等于主线程的 PID；同一进程内多个线程共享同一个 TGID。

用户态看进程时，通常看到的是线程组；看 `/proc/<pid>/task/` 时能看到该进程下的各个线程任务。

面试时不必展开太深，但一句“Linux 用 task 表示执行实体，通过 clone 的共享标志区分进程/线程风格”很加分。

## 9. 多进程和多线程如何选择

### 9.1 多进程适合什么

多进程适合：

- 需要强隔离，避免一个任务崩溃影响全局。
- 需要利用多核，同时希望内存隔离简单。
- 运行不同程序或不同权限的任务。
- 服务模型天然独立，例如 master-worker、多进程 Web 服务。
- 绕开某些语言运行时限制，例如 Python CPU 密集任务绕开 GIL。

缺点是：创建和切换成本更高，通信更麻烦，数据共享需要 IPC。

### 9.2 多线程适合什么

多线程适合：

- 同一进程内部有多个并发任务。
- 任务之间需要共享大量内存状态。
- I/O 等待较多，希望一个线程等待时另一个继续执行。
- 需要在多核上并行执行计算，且语言运行时支持真正并行。

缺点是：共享内存带来数据竞争、死锁、活锁、调试困难等问题。

### 9.3 多线程一定比多进程快吗

不一定。

多线程的优势主要来自共享资源和较低切换成本，但真实性能还取决于：

- 是否 CPU 密集还是 I/O 密集。
- 是否有锁竞争。
- 是否频繁上下文切换。
- 是否造成缓存失效和伪共享。
- 语言运行时是否限制并行，例如 CPython 的 GIL。
- 任务之间是否需要隔离和容错。

更稳的回答：线程通常更轻，但不是越多越快。线程数超过 CPU 核心太多，或者大量线程都在抢同一把锁，性能可能下降。

## 10. 进程状态和线程状态

操作系统会把执行实体放入不同状态队列。常见状态：

| 状态 | 含义 |
|---|---|
| New | 正在创建 |
| Ready | 已准备好，等待 CPU |
| Running | 正在 CPU 上运行 |
| Blocked/Waiting | 等待 I/O、锁、事件等 |
| Terminated | 已结束 |

进程和线程都可能经历类似状态。面试中如果追问“阻塞一个线程会不会阻塞整个进程”，要分情况：

- 用户级线程如果完全由用户态库调度，一个线程发起阻塞系统调用可能影响整个进程。
- 现代主流 OS 的内核级线程中，一个线程阻塞，通常不妨碍同进程其他可运行线程被调度。

Linux pthreads 属于内核可感知的线程模型，阻塞一个线程不等于阻塞整个进程。

## 11. 用户线程、内核线程、协程

### 11.1 用户级线程

用户级线程由用户态库管理，切换快，不一定需要进入内核。但内核不知道这些线程的存在，阻塞系统调用和多核调度会比较麻烦。

### 11.2 内核级线程

内核级线程由操作系统调度，能更好利用多核；一个线程阻塞时，内核可以调度同进程其他线程。代价是创建和切换要经过内核，成本更高。

### 11.3 协程

协程通常是用户态的轻量执行单元，由程序或运行时主动调度。它适合大量 I/O 并发，例如异步网络服务。

进程、线程、协程可以这样记：

| 概念 | 调度者 | 切换成本 | 典型用途 |
|---|---|---|---|
| 进程 | 操作系统 | 高 | 隔离、容错、多程序 |
| 线程 | 操作系统或线程库 | 中 | 并发执行、共享内存 |
| 协程 | 用户态运行时 | 低 | 高并发 I/O、异步任务 |

注意：协程不是操作系统资源分配单位，也不是传统意义上的 CPU 调度基本单位。它通常最终还是跑在线程之上。

## 12. 经典追问

### 12.1 进程崩溃会不会影响其他进程

通常不会直接影响，因为地址空间隔离。一个进程非法访问内存，操作系统可以只终止这个进程。

但也有间接影响：

- 它持有的锁、文件、共享内存可能影响其他进程。
- 它是父进程或服务主进程时，退出可能导致子进程或服务不可用。
- 它写坏了共享存储或外部状态，其他进程仍会受影响。

### 12.2 一个线程崩溃会不会影响整个进程

通常会。因为线程共享进程地址空间，一个线程的非法写内存可能破坏全局状态；未处理的致命信号也可能终止整个进程。

### 12.3 为什么浏览器常用多进程

现代浏览器常把页面、插件、GPU、网络等拆到不同进程，核心原因是隔离和安全：

- 一个页面崩溃不至于整个浏览器崩溃。
- 不同站点可以放入不同沙箱，降低安全风险。
- 多进程边界更适合权限控制。

代价是内存占用和 IPC 成本更高。

### 12.4 为什么线程通信容易但更危险

因为线程共享地址空间，读写同一对象很方便；但这种方便也意味着任何线程都可能写坏共享状态。共享内存并发的核心难点不是“怎么传数据”，而是“怎么保证数据一致”。

### 12.5 `fork()` 后父子进程共享内存吗

从语义上看，父子进程有各自独立的地址空间。现代系统通常用写时复制优化：刚 `fork()` 后物理页可能暂时共享，直到某一方写入时再复制。

所以面试回答要区分“逻辑语义”和“内核优化”：逻辑上独立，物理上可能短暂共享只读页。

## 13. 面试标准答案

### 13.1 30 秒版本

进程是资源分配和隔离的基本单位，线程是进程内部的执行流，也是现代操作系统中 CPU 调度的基本单位。进程之间通常有独立地址空间，通信需要 IPC；同一进程内线程共享地址空间、堆、全局变量、文件描述符等资源，但每个线程有自己的栈、寄存器和程序计数器。线程创建和切换通常比进程轻，但共享内存也带来数据竞争和同步问题。

### 13.2 1 分钟版本

进程可以理解为运行中的程序实例，它拥有独立的虚拟地址空间和系统资源，比如打开文件、Socket、环境变量、权限信息等。线程是进程中的一条执行流，同一个进程可以有多个线程并发执行。

进程之间隔离性更好，一个进程崩溃通常不直接影响另一个进程，但进程通信需要管道、消息队列、共享内存、Socket 等 IPC。线程之间共享同一地址空间，所以通信方便，创建和切换成本通常更低，但需要用互斥锁、条件变量、信号量、原子操作等机制解决并发安全问题。

在 Linux 中，进程和线程都用 task 表示，线程通常是通过 `clone()` 创建的、共享地址空间等资源的轻量级进程。

## 14. 常见误区

- **误区 1：只背“进程资源分配，线程 CPU 调度”。** 这只是入口，必须补充地址空间、资源共享和同步通信。
- **误区 2：说线程共享栈。** 同一进程内线程共享堆和全局变量，但每个线程有自己的栈。
- **误区 3：说多线程一定更快。** 锁竞争、上下文切换、缓存抖动、GIL 都可能让多线程变慢。
- **误区 4：说进程之间完全不能共享内存。** 默认地址空间独立，但可以通过共享内存、mmap 等 IPC 主动共享。
- **误区 5：把并发和并行混为一谈。** 并发是多个任务交替推进；并行是多个任务同一时刻在多个核心上运行。
- **误区 6：把 Linux 线程理解成完全独立的一种内核对象。** Linux 更准确的理解是 task + 资源共享标志。

## 我应该背下来的最短答案

进程是运行中的程序实例，是资源分配和隔离的基本单位；线程是进程内的执行流，是 CPU 调度的基本单位。进程有独立地址空间和系统资源，进程间通信要靠 IPC；同一进程的线程共享地址空间、堆、全局变量、文件描述符等资源，但各自有栈、寄存器和程序计数器。线程通常创建和切换更轻，但共享内存会带来竞态条件，需要锁、信号量、条件变量或原子操作同步。Linux 中线程本质上是通过 `clone()` 创建的、共享更多资源的轻量级 task。
