# 关系型和非关系型数据库区别

## 1. 面试官到底想考什么

面试官想看你是否理解不同数据库适合不同场景。不是所有数据都适合放 MySQL，也不是非关系型数据库就一定比关系型数据库高级。

这题的核心是：**数据结构是否固定、是否需要事务、查询方式是什么、扩展方式是什么、适合什么业务场景**。

## 2. 零基础版解释

关系型数据库像 Excel 表格：每一列是什么含义提前定好，比如用户表有 id、姓名、手机号、创建时间。每一行都是一个用户。它很适合结构清楚、规则明确的数据。

非关系型数据库更像一个文件柜：有的抽屉放 JSON 文档，有的放键值对，有的放图关系。它不一定要求每条数据都长得一样，所以更灵活。

如果你要管理银行账户、订单、库存，通常更适合关系型数据库，因为这些数据要求准确、一致、可事务回滚。如果你要存缓存、日志、用户行为、灵活变化的 JSON 文档，非关系型数据库可能更合适。

## 3. 正式概念

**关系型数据库** 使用表结构组织数据，通过行和列表达实体及属性，使用 SQL 查询，通常强调事务、一致性和复杂查询能力。典型代表是 MySQL、PostgreSQL、Oracle。

**非关系型数据库** 不以传统二维表作为主要模型，可能使用键值、文档、列族、图等数据模型，通常强调灵活 schema、水平扩展和特定场景性能。典型代表是 Redis、MongoDB、Cassandra、Neo4j。

| 维度 | 关系型数据库 | 非关系型数据库 |
|---|---|---|
| 数据模型 | 表、行、列 | 键值、文档、列族、图等 |
| Schema | 通常固定 | 通常更灵活 |
| 查询语言 | SQL | 各产品不同 |
| 事务能力 | 通常较强 | 视产品而定 |
| 扩展方式 | 常见为纵向扩展，也可分库分表 | 常见为水平扩展 |
| 典型场景 | 订单、支付、用户、库存 | 缓存、日志、文档、推荐关系 |

### 3.1 关系型数据库为什么叫“关系型”

这里的“关系”不是人际关系，而是数学里的关系模型。简单理解就是：一张二维表表示一类实体，表与表之间可以通过字段建立关联。

例如电商系统里可能有三张表：

```text
用户表 user(id, name, phone)
订单表 order(id, user_id, total_price)
商品表 product(id, name, price)
```

订单表里的 `user_id` 可以关联用户表的 `id`。这样就能表达“哪个用户下了哪个订单”。

关系型数据库擅长处理：

- 数据结构明确。
- 数据之间关系清楚。
- 需要复杂查询。
- 需要事务保证一致性。

### 3.2 SQL 是什么

SQL 是关系型数据库常用的查询语言。它关注“我要什么数据”，而不是一步步告诉数据库怎么取。

例如：

```sql
SELECT id, name
FROM user
WHERE phone = '13800000000';
```

这条 SQL 的意思是：从用户表中查手机号为指定值的用户 id 和姓名。

数据库内部会根据表结构、索引、统计信息等决定具体执行方式。

### 3.3 事务和 ACID

关系型数据库的重要能力是事务。事务可以把多条操作看成一个整体，要么都成功，要么都失败。

经典例子是转账：

```text
A 账户扣 100
B 账户加 100
```

这两步必须一起成功。如果 A 扣了钱，B 没加上，数据就错了。

事务的 ACID：

| 特性 | 含义 | 人话解释 |
|---|---|---|
| Atomicity 原子性 | 事务不可分割 | 要么全做，要么全不做 |
| Consistency 一致性 | 数据从一个正确状态到另一个正确状态 | 转账前后总金额不变 |
| Isolation 隔离性 | 并发事务互不干扰 | 多个人同时操作也不能乱 |
| Durability 持久性 | 提交后数据不会丢 | 成功提交后落盘保存 |

### 3.4 非关系型数据库有哪些类型

非关系型数据库不是一种数据库，而是一类数据库。

| 类型 | 代表 | 数据模型 | 适合场景 |
|---|---|---|---|
| 键值数据库 | Redis | key-value | 缓存、计数器、会话 |
| 文档数据库 | MongoDB | JSON/BSON 文档 | 内容管理、灵活字段 |
| 列族数据库 | Cassandra、HBase | 按列族组织 | 海量写入、宽表 |
| 图数据库 | Neo4j | 点和边 | 社交关系、路径查询 |
| 搜索引擎 | Elasticsearch | 倒排索引文档 | 全文搜索、日志检索 |

面试中不要把“非关系型数据库”简单等同于 Redis。Redis 只是其中一种。

### 3.5 如何选择数据库

可以按这几个问题判断：

1. 数据结构是否稳定？
2. 是否需要复杂 SQL 查询？
3. 是否需要强事务？
4. 读写量有多大？
5. 是否主要做缓存或临时数据？
6. 是否需要全文搜索、图关系、海量日志等特殊能力？

典型选择：

- 核心交易数据：优先关系型数据库。
- 热点缓存：Redis。
- 灵活 JSON 文档：MongoDB。
- 全文检索：Elasticsearch。
- 复杂关系路径：图数据库。

### 3.6 一致性和扩展性的取舍

关系型数据库通常更重视强一致性。非关系型数据库在大规模分布式场景下，可能更强调可用性和扩展性，有时接受最终一致性。

**最终一致性**的意思是：短时间内不同节点看到的数据可能不完全一致，但经过同步后最终会一致。

例如点赞数、浏览量、日志统计等场景，短暂不一致通常可以接受；但支付余额、库存扣减等场景，通常不能随便接受不一致。

## 4. 面试回答模板

关系型数据库和非关系型数据库最大的区别在于数据组织方式和适用场景。

关系型数据库用表来组织数据，表结构比较固定，通常使用 SQL 查询，支持比较强的事务能力和复杂查询能力。比如 MySQL、PostgreSQL 很适合订单、用户、支付这类结构清晰、对一致性要求高的业务。

非关系型数据库不一定使用二维表，它可能是键值型、文档型、列族型或图数据库。它的 schema 通常更灵活，也更强调水平扩展和特定场景性能。比如 Redis 常用于缓存，MongoDB 常用于文档数据，Neo4j 常用于图关系。

所以不是谁替代谁，而是看业务需求。如果数据关系明确、需要事务和复杂查询，我会优先考虑关系型数据库；如果数据结构变化大、访问模式明确、需要高并发缓存或灵活扩展，可以考虑非关系型数据库。

## 6. 常见误区

- **误区 1：说非关系型数据库一定比关系型数据库快。** 快不快取决于场景、数据模型、索引和访问模式。
- **误区 2：说非关系型数据库不支持事务。** 一些非关系型数据库也提供事务能力，只是模型和使用方式不同。
- **误区 3：只讲定义不讲场景。** 面试中一定要结合订单、缓存、日志、文档等场景。
- **误区 4：把 Redis 当成普通数据库替代 MySQL。** Redis 常用于缓存和高性能键值访问，不适合替代所有持久化关系数据。

## 7. 记忆口诀

**表格查关系，文档键值求灵活；强事务看关系，高扩展看场景。**

## 我应该背下来的最短答案

关系型数据库用表组织数据，schema 固定，SQL 查询能力强，事务能力强，适合订单、支付、用户等结构化数据。非关系型数据库数据模型更灵活，比如键值、文档、图，适合缓存、日志、文档和高扩展场景。选择时主要看数据结构、事务要求、查询方式和扩展需求。
