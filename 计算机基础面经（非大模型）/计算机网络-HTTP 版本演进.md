# HTTP 版本演进

## 1. 面试官到底想考什么

这题表面问“HTTP 有几个版本”，实际考你是否知道 Web 请求为什么越来越快。面试官希望你能说出每个版本解决了什么问题，而不是只背版本号。

重点不是年份，而是演进逻辑：**连接复用、并发能力、头部压缩、传输层变化**。

## 2. 零基础版解释

可以把浏览网页想成去餐厅点菜。

HTTP/1.0 像是你点一道菜，服务员跑一趟；再点一道，服务员再跑一趟。每次都重新建立联系，效率低。

HTTP/1.1 像是服务员一直站在你桌边，你可以连续点菜，不用每次重新叫人。但如果前一道菜卡住，后面的菜也可能受影响。

HTTP/2 像是服务员一次帮你处理多道菜的请求，而且还把菜单信息压缩得更短，效率更高。

HTTP/3 更进一步，不再依赖原来的 TCP 通道，而是用 QUIC，让连接建立更快，也减少某些阻塞问题。

## 3. 正式概念

HTTP 是应用层协议，用于客户端和服务器之间传输超文本、JSON、图片、文件等资源。

常见版本：

| 版本 | 核心特点 | 主要问题或改进 |
|---|---|---|
| HTTP/0.9 | 只支持简单 GET | 功能非常简单 |
| HTTP/1.0 | 引入请求头、响应头、状态码 | 默认短连接，连接开销大 |
| HTTP/1.1 | 默认长连接，支持 Host、缓存、分块传输 | 同一连接上仍有队头阻塞 |
| HTTP/2 | 二进制分帧、多路复用、头部压缩、服务器推送 | 基于 TCP，TCP 层丢包仍可能影响整体 |
| HTTP/3 | 基于 QUIC，运行在 UDP 之上 | 部署和兼容性更复杂 |

### 3.1 HTTP 请求和响应长什么样

HTTP 是典型的**请求-响应模型**。客户端发请求，服务端返回响应。

一个 HTTP 请求通常包括：

```http
GET /users/123 HTTP/1.1
Host: example.com
Accept-Language: zh-CN
Accept: application/json
```

一个 HTTP 响应通常包括：

```http
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 28

{"id":123,"name":"Alice"}
```

HTTP 报文大致由三部分组成：

- **起始行**：请求行或状态行。
- **头部字段**：描述请求、响应、缓存、认证、内容类型等元信息。
- **消息体**：真正传输的数据，例如 JSON、HTML、图片内容。

### 3.2 常见 HTTP 方法

| 方法      | 语义      | 常见用途      |
| ------- | ------- | --------- |
| GET     | 获取资源    | 查询数据、打开页面 |
| POST    | 提交数据    | 创建资源、提交表单 |
| PUT     | 整体更新资源  | 替换某个对象    |
| PATCH   | 局部更新资源  | 修改对象部分字段  |
| DELETE  | 删除资源    | 删除数据      |
| HEAD    | 只获取响应头  | 检查资源元信息   |
| OPTIONS | 查询支持的方法 | 跨域预检请求    |

面试中注意：GET 和 POST 不是“一个安全一个不安全”的区别。它们的核心区别是**语义、参数位置习惯、幂等性预期和缓存行为不同**。安全性主要取决于是否使用 HTTPS、服务端如何处理数据。

### 3.3 常见状态码

| 状态码 | 含义                    | 面试解释       |
| --- | --------------------- | ---------- |
| 200 | OK                    | 请求成功       |
| 201 | Created               | 创建成功       |
| 301 | Moved Permanently     | 永久重定向      |
| 302 | Found                 | 临时重定向      |
| 304 | Not Modified          | 缓存可用，资源未修改 |
| 400 | Bad Request           | 请求格式或参数错误  |
| 401 | Unauthorized          | 未认证        |
| 403 | Forbidden             | 已认证但无权限    |
| 404 | Not Found             | 资源不存在      |
| 500 | Internal Server Error | 服务端内部错误    |
| 502 | Bad Gateway           | 网关收到上游错误响应 |
| 503 | Service Unavailable   | 服务暂不可用     |
| 504 | Gateway Timeout       | 网关等待上游超时   |

状态码可以按首位记：

- **2xx**：成功。
- **3xx**：重定向或缓存。
- **4xx**：客户端请求问题。
- **5xx**：服务端或网关问题。

### 3.4 HTTP/1.1 的长连接

HTTP/1.0 默认短连接。每请求一个资源都要建立 TCP 连接，请求完成后关闭连接。

问题是，一个网页可能包含 HTML、CSS、JS、图片等很多资源。如果每个资源都重新建立 TCP 连接，三次握手和慢启动都会带来额外开销。

HTTP/1.1 默认使用长连接：

```http
Connection: keep-alive
```

长连接的好处：

- 减少 TCP 建连和断连成本。
- 多个请求可以复用同一条连接。
- 提高页面加载效率。

但 HTTP/1.1 仍然可能有队头阻塞：同一连接中，如果前面的响应迟迟不返回，后面的响应也可能被挡住。

### 3.5 HTTP/2 的二进制分帧和多路复用

HTTP/2 不再像 HTTP/1.1 那样以纯文本方式传输报文，而是拆成更小的二进制帧。

几个关键概念：

- **帧**：HTTP/2 传输的最小单位。
- **流**：一个请求-响应对应一个流。
- **多路复用**：同一条 TCP 连接上可以同时传输多个流的帧。

好处是：多个请求不用排队等前一个完整响应结束，可以交错传输。

但是 HTTP/2 仍基于 TCP。TCP 层如果发生丢包，后续字节必须等待重传，因此仍可能出现 TCP 层面的队头阻塞。

### 3.6 HTTP/3 和 QUIC

HTTP/3 使用 QUIC 作为传输层基础。QUIC 通常运行在 UDP 之上，但它自己实现了类似 TCP 的可靠传输、拥塞控制和加密能力。

HTTP/3 的目标：

- 减少连接建立延迟。
- 改善 TCP 层队头阻塞问题。
- 更好支持连接迁移，例如手机从 Wi-Fi 切到移动网络。

可以这样理解：HTTP/3 不是简单“HTTP 换成 UDP”，而是 HTTP 跑在 QUIC 上，QUIC 借 UDP 承载，并在用户态实现可靠传输和安全能力。

### 3.7 HTTP 和 HTTPS

HTTPS 不是新的应用层语义协议，而是：

```text
HTTP + TLS
```

HTTPS 主要提供：

- **加密**：防止内容被窃听。
- **身份认证**：通过证书确认服务端身份。
- **完整性保护**：防止数据被篡改。

面试中如果被问 HTTP 和 HTTPS 的区别，可以从安全能力、端口、证书、握手开销几个角度回答。

## 4. 面试回答模板

HTTP 是应用层协议，主要用于客户端和服务端之间传输资源。常见版本可以按演进来讲。

HTTP/1.0 已经有请求头、响应头和状态码，但默认是短连接，每次请求都要建立和关闭连接，开销比较大。

HTTP/1.1 默认使用长连接，可以在一个 TCP 连接上发送多个请求，还支持 Host 头、缓存控制、分块传输等机制。但它的问题是同一连接上的请求可能发生队头阻塞。

HTTP/2 做了比较大的优化，它使用二进制分帧，支持多路复用，也就是一个连接里可以并发传多个请求和响应；同时支持头部压缩，减少重复 header 的传输开销。

HTTP/3 则把底层从 TCP 换成了基于 UDP 的 QUIC，目标是减少连接建立延迟，并改善 TCP 层丢包导致的阻塞问题。

所以 HTTP 的演进主线就是：从短连接到长连接，从文本到二进制分帧，从串行到多路复用，再到基于 QUIC 优化传输延迟。

## 6. 常见误区

- **误区 1：只背版本号。** 面试中更重要的是讲清楚每个版本解决的问题。
- **误区 2：以为 HTTP/2 完全解决所有队头阻塞。** HTTP/2 解决了 HTTP 层队头阻塞，但基于 TCP 时仍可能受 TCP 层丢包影响。
- **误区 3：说 HTTP/3 是基于 TCP。** HTTP/3 基于 QUIC，QUIC 通常运行在 UDP 之上。
- **误区 4：把 HTTP 和 HTTPS 当成两个完全不同协议。** HTTPS 本质是在 HTTP 和传输层之间加了 TLS 加密。

## 7. 记忆口诀

**一点一长连，二路多复用，三换 QUIC 跑更快。**

解释：

- HTTP/1.1：长连接
- HTTP/2：多路复用、头部压缩
- HTTP/3：QUIC
