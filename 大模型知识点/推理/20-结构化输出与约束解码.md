# 结构化输出与约束解码

> 要求模型输出 JSON 时，仅在 prompt 中写“请输出 JSON”不能保证语法合法。约束解码在每一步屏蔽会使当前前缀无法匹配目标语法的 token。

## 1. 基本过程

设当前语法状态为 $`s_t`$，允许 token 集合为 $`A(s_t)`$：

```math
z_i'=\begin{cases}
z_i,&i\in A(s_t),\\
-\infty,&i\notin A(s_t).
\end{cases}
```

采样 token 后，解析器转移到新状态：

```math
s_{t+1}=\delta(s_t,x_{t+1}).
```

只要 grammar 与 tokenizer 处理正确，生成结果在语法上有效。

## 2. Regex、FSM 与 CFG

- **Regex/FSM**：适合正则语言，状态转移快速；
- **CFG**：支持嵌套括号、递归 JSON/代码结构；
- **JSON Schema**：需转换为 grammar，并处理类型、required、enum 等；
- **Choice**：有限字符串集合，可构建 trie。

语义条件如“结束日期必须晚于开始日期”通常无法只靠 CFG 保证。

## 3. Tokenizer 带来的困难

语法基于字符/字节，模型基于 token。一个 token 可能包含：

- 多个字符；
- 半个 UTF-8 字符边界；
- 引号、逗号与普通文本的组合；
- 前导空格。

解析器必须判断整个 token 字节串是否能从当前状态合法消费，不能假设“一 token 一字符”。

## 4. Mask 生成成本

朴素做法每步遍历整个词表，CPU 逐 token 检查 grammar，可能拖慢 decode。优化包括：

- 预计算状态到 token bitmap；
- trie/automaton；
- 压缩 FSM；
- batch 中按 grammar/state 分组；
- GPU 上应用 bitmask；
- 异步准备下一步 mask。

## 5. Grammar Compilation Cache

相同 JSON Schema 可能被大量请求复用。把 schema 编译成 FSM/CFG automaton 后缓存，可避免重复编译。Cache key 必须包含 tokenizer 和 parser 版本。

## 6. 与采样的关系

约束解码先删除非法 token，再在合法集合上执行 temperature/top-p 等采样。若先 top-p 再约束，可能把所有合法 token 都截掉。

正确顺序通常是：

```text
logits processors → grammar mask → renormalize/sample
```

具体惩罚和 mask 的顺序应由语义明确规定。

## 7. Tool Calling

工具调用通常包含：

- 选择工具名；
- 生成符合该工具 schema 的 arguments；
- 解析模型特定 tool-call token；
- 流式拼接部分 JSON；
- 校验与执行前安全检查。

语法合法不等于工具参数安全，仍需业务校验和权限控制。

## 8. 停止条件

Grammar 到达接受状态不一定必须立即停止；有的语法允许继续。运行时需要区分：

- accepting state；
- only accepting/terminal state；
- EOS 是否在允许集合。

## 9. 评测

- syntactic validity；
- schema validity；
- semantic/task accuracy；
- grammar compile latency；
- mask latency/token；
- 与无约束基线的 TPOT 差异；
- batch 中多种 schema 的吞吐。

## 10. 面试回答

> 约束解码把 regex、JSON Schema 或 CFG 编译成自动机。每一步根据当前状态得到合法 token 集，将其他 logits 设为负无穷，采样后更新状态。难点是 tokenizer token 可能包含多个字节、CFG 状态复杂，以及逐 token mask 的 CPU 开销；常通过预计算 bitmap、grammar cache、压缩状态机和 GPU mask 优化。
