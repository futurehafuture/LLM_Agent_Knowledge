# 分布式推理与模型并行

> 单卡放不下模型，或一张卡的吞吐与延迟无法满足目标时，需要在多 GPU、多节点之间切分权重、请求或序列。

## 1. 先区分扩容目标

- **容量扩展**：让模型能够放下；
- **单请求加速**：降低一个请求的延迟；
- **吞吐扩展**：服务更多独立请求；
- **长上下文扩展**：分摊单序列的 attention/KV；
- **MoE 扩展**：把不同 experts 分布到设备。

不同并行方式对应不同目标。

## 2. Data Parallelism

每个 replica 保存完整模型，路由不同请求：

```math
\text{Throughput}_{DP}\approx R\cdot\text{Throughput}_{single}
```

理想情况下可近线性扩展，不需要每层通信；但单卡必须放得下模型，单请求延迟不会降低。生产服务最优先考虑 DP，因为故障隔离和扩缩容简单。

## 3. Tensor Parallelism

把一个矩阵乘沿维度切到多卡。以列切分为例：

```math
W=[W_1,W_2,\dots,W_P],\quad Y_i=XW_i.
```

后续可能需要 AllGather 或 AllReduce 合并结果。每层都会通信，因此 TP 更适合节点内 NVLink/NVSwitch。

通信时间粗略表示为：

```math
T_{comm}\approx\alpha+\frac{S}{BW_{link}},
```

其中 $`\alpha`$ 是启动延迟，$`S`$ 是消息大小。Decode 的小消息尤其容易受 $`\alpha`$ 影响。

## 4. Pipeline Parallelism

按层切分模型：GPU 0 运行前几层，GPU 1 运行后几层。单个 microbatch 必须依次经过各 stage，会产生 pipeline bubble。

服务中可通过多个并发 microbatch 填充流水线，但单请求延迟会叠加各 stage 与通信时间。PP 适合模型跨节点放置或 TP 域不宜继续扩大时。

## 5. Context / Sequence Parallelism

把同一个长序列的 token 或 KV 沿序列维切分。每个设备保存部分上下文，attention 需要通过 ring、all-to-all 或分层通信获得全局结果。

它主要解决超长上下文的显存和 attention 计算，短上下文下通信成本可能不划算。

## 6. Expert Parallelism

MoE 中不同 experts 放在不同设备。Router 决定 token 去向，再通过 All-to-All 发送 token：

```text
tokens → router → all-to-all → local experts → all-to-all → original order
```

难点是负载不均衡、跨节点带宽和小批次下的低效 GEMM。

## 7. 并行组合

大型部署常写成：

```math
N_{GPU}=N_{DP}\times N_{PP}\times N_{TP}\times N_{EP}\times N_{CP}.
```

例如单节点 8 卡做 TP=8，多个节点作为 DP replicas；或节点内 TP、节点间 PP/EP。

## 8. 如何选择

1. 模型单卡放得下：优先 DP，最简单、吞吐扩展最好。
2. 单卡放不下但单节点放得下：节点内 TP。
3. 模型跨节点：TP 域尽量留在高速互联内，节点间考虑 PP。
4. MoE：根据 expert 数和网络布局配置 EP。
5. 超长上下文：再引入 CP/SP。

## 9. Prefill 与 Decode 的并行差异

- Prefill GEMM 大，TP 的计算缩放更容易覆盖通信；
- Decode 每层计算小、同步频繁，扩大 TP 可能让通信占比上升；
- 因此同一模型在两个阶段的最佳并行配置可能不同，这也是 PD 分离的动机之一。

## 10. 通信优化

- 通信与计算重叠；
- fused collective + GEMM；
- 拓扑感知 rank mapping；
- 节点内 NVLink、节点间 InfiniBand 分层 collective；
- 减少 TP 域，改用更多 DP；
- CUDA Graph 固定通信路径。

## 11. 面试回答

> DP 复制完整模型，几乎不增加层间通信，适合扩吞吐；TP 切矩阵，每层需要 collective，适合节点内高速互联并能降低单请求计算；PP 按层切模型，容量扩展好但有流水线气泡；CP 切长序列；EP 将 MoE experts 分布到设备并依赖 All-to-All。选型必须结合模型能否放下、网络拓扑、Prefill/Decode 比例和延迟目标。
