# 推理服务架构与生产部署

> 模型 kernel 只决定部分性能。生产系统还需要路由、排队、限流、扩缩容、版本管理、流式连接和故障恢复。

## 1. 分层架构

```text
Client
  → API Gateway / Auth / Rate Limit
  → Global Router
  → Model/Adapter Router
  → Replica Scheduler
  → Inference Engine
  → GPU Workers
  → Metrics / Tracing / Billing
```

每层都可能增加延迟，也都承担必要的隔离职责。

## 2. Router 需要知道什么

- 模型与版本；
- adapter；
- 请求输入长度、最大输出长度；
- 多模态资源；
- replica 队列、KV 容量和 cache locality；
- TP/PP/EP group 健康状态；
- 优先级和租户配额；
- PD 分离拓扑。

只按“最少连接数”路由通常不足以反映 GPU 真实负载。

## 3. 准入与 Backpressure

系统应在资源不足时明确排队、降级或拒绝，而不是无限积压直到 OOM。

Little 定律：

```math
L=\lambda W,
```

到达率 $`\lambda`$ 接近服务能力时，等待时间 $`W`$ 上升会使在途请求数 $`L`$ 急剧增长，进一步占用内存与连接。

## 4. 限流维度

仅按 RPS 限流不公平。一个 100K 输入、10K 输出的请求远大于短对话。可组合：

- requests/min；
- input/output tokens/min；
- concurrent sequences；
- GPU-seconds；
- image pixels/video seconds；
- adapter/model 权重。

## 5. Autoscaling

GPU Pod 启动慢、模型加载大，不能只根据瞬时 GPU utilization 扩容。常见信号：

- queue depth / queue delay；
- token arrival rate；
- KV usage；
- TTFT/TPOT SLO；
- prefix cache hit；
- 预测的 workload。

需要预热新 replica，包括权重加载、kernel autotune 和 CUDA Graph capture。

## 6. 流式输出与取消

客户端断开后必须把取消传播到 scheduler，释放：

- waiting/running request；
- KV blocks；
- speculative buffers；
- encoder work；
- 网络流和计费状态。

若只关闭 HTTP 连接而 GPU 继续生成，会浪费大量算力。

## 7. 版本与灰度

- immutable model version；
- adapter version；
- tokenizer/template version；
- canary traffic；
- shadow testing；
- 质量、延迟和错误率联合回滚。

Prefix/KV cache 必须按版本隔离，不能在权重变化后继续复用。

## 8. 故障模型

- 单 worker crash；
- 一个 TP rank 失败导致整个 group 失败；
- 节点或网络故障；
- NCCL collective hang；
- OOM；
- 模型加载失败；
- 流式连接中断。

对多卡 group，健康检查必须覆盖所有 ranks。恢复可以重试、重放或迁移，但生成中的随机状态与 KV 使“无缝迁移”并不简单。

## 9. 可观测性

至少采集：

- 请求：TTFT、TPOT、E2E、状态码、取消；
- Scheduler：waiting/running、batch tokens、preemption；
- Cache：KV usage、prefix hit、eviction；
- GPU：显存、SM、HBM、功耗、ECC；
- 网络：NCCL/RDMA throughput、重传；
- 业务：模型/租户/adapter/长度分布。

高基数字段不要直接作为 Prometheus label，可放 trace/log。

## 10. 成本

```math
\text{Cost per 1M output tokens}
=\frac{\text{GPU hourly price}\times\text{GPU count}}{\text{output TPS}\times3600}\times10^6.
```

还应计入空闲副本、失败请求、输入 token、网络、CPU 和存储。低利用率但低延迟的在线服务与高利用率离线批处理成本结构不同。

## 11. 面试回答

> 生产推理服务由网关、模型路由、scheduler、GPU worker 和可观测性组成。核心不是无限堆请求，而是按 token/KV 预算做准入和 backpressure，用 queue delay、SLO 与 token rate 扩缩容。请求取消要释放 GPU 状态，版本变化要隔离 KV cache，多卡 group 的任一 rank 故障都可能让整个 replica 不可用。
