# BF16、FP16 与 FP32：浮点精度详解

> 核心问题：为什么大模型训练用 BF16 而不是 FP16？混合精度训练的原理是什么？

---

## 1. 三种格式的位结构

IEEE 754 浮点数 = **符号位(Sign) + 指数位(Exponent) + 尾数位(Mantissa)**

```
FP32 (float32):  1位符号 + 8位指数 + 23位尾数 = 32 bit
FP16 (float16):  1位符号 + 5位指数 + 10位尾数 = 16 bit
BF16 (bfloat16): 1位符号 + 8位指数 +  7位尾数 = 16 bit
```

| 属性 | FP32 | FP16 | BF16 |
|---|---|---|---|
| 总位数 | 32 | 16 | 16 |
| 指数位 | 8 | **5** | **8** |
| 尾数位 | 23 | 10 | **7** |
| 动态范围 | ±3.4e38 | ±65504 | **±3.4e38** |
| 精度（十进制有效位） | ~7 位 | ~3 位 | ~2 位 |
| 最小正值 | 1.2e-38 | 6.1e-5 | 1.2e-38 |
| 显存占用 | 1× | 0.5× | 0.5× |

> [!tip] 一句话记忆
> BF16 = FP32 的指数位（保持动态范围） + 砍掉尾数位（降低精度）。
> FP16 = FP32 砍掉指数位和尾数位（动态范围和精度都降低）。

---

## 2. 为什么 BF16 优于 FP16？

### 动态范围（指数位决定）

| | 指数位 | 可表示范围 | 能否覆盖 loss 值 |
|---|---|---|---|
| FP32 | 8 | ±3.4e38 | 完全覆盖 |
| FP16 | 5 | ±65504 | **不够**！loss 稍大就溢出 |
| BF16 | 8 | ±3.4e38 | 和 FP32 一样 |

FP16 的指数位只有 5 位，动态范围只有 $2^{-14} \sim 65504$。大模型训练中：
- Loss 值可能 > 65504 → **上溢出（overflow）**
- 梯度可能 < $6.1 \times 10^{-5}$ → **下溢出（underflow）**

BF16 保留了 FP32 的 8 位指数，动态范围完全一致，不会溢出。

### 精度（尾数位决定）

| | 尾数位 | 精度 |
|---|---|---|
| FP32 | 23 | ~7 位十进制 |
| FP16 | 10 | ~3 位十进制 |
| BF16 | 7 | ~2 位十进制 |

BF16 精度最低（只有 2-3 位有效数字），但对于神经网络来说：
- 模型参数和梯度本身就是噪声估计，不需要极高精度
- 指数范围比精度重要得多——溢出是致命的，精度低一点可以接受

### FP32 → BF16 的转换极其简单

```python
# BF16 本质上就是 FP32 截断高 16 位
# FP32: [S][EEEEEEEE][MMMMMMMMMMMMMMMMMMMMMMM]  (1+8+23)
# BF16: [S][EEEEEEEE][MMMMMMM]                    (1+8+7)
#                    ↑ 保留     ↑ 截断
```

在硬件上只需要一次位操作，不需要舍入逻辑，转换代价接近零。

## 3. 混合精度训练（Mixed Precision Training）

### 为什么不全用低精度？

1. **前向/反向**：可以用 BF16/FP16（省显存、加速）
2. **参数更新**：必须在 FP32 下做（精度不够会导致训练不稳定）

### 混合精度训练流程

```
┌─────────────────────────────────────────────────────┐
│  1. 维护一份 FP32 master weights（主权重）            │
│  2. 将 master weights 转成 BF16 → 前向传播            │
│  3. BF16 反向传播得到 BF16 梯度                       │
│  4. 梯度转回 FP32                                     │
│  5. 在 FP32 下更新 master weights                     │
│  6. 下一步重复 2-5                                     │
└─────────────────────────────────────────────────────┘
```

```python
import torch

model = MyModel().cuda()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

# 使用 PyTorch 原生混合精度
scaler = torch.amp.GradScaler("cuda")  # FP16 需要；BF16 不需要 scaler

for data, target in dataloader:
    optimizer.zero_grad()

    with torch.amp.autocast("cuda", dtype=torch.bfloat16):
        output = model(data)
        loss = criterion(output, target)

    # BF16 不需要 scaler；FP16 需要 scaler.scale(loss).backward()
    loss.backward()
    optimizer.step()
```

### GradScaler 的作用（FP16 专属）

FP16 梯度容易下溢出，GradScaler 做两件事：
1. **Loss Scaling**：反向传播前把 loss 乘一个大数（如 65536），防止小梯度下溢
2. **Unscaling**：梯度更新前除回来

BF16 不需要 GradScaler，因为动态范围足够。

## 4. 不同场景的精度选择

| 场景 | 推荐精度 | 原因 |
|---|---|---|
| 预训练 | BF16 混合精度 | 动态范围够，不需要 scaler，训练稳定 |
| 微调（LoRA等） | BF16 混合精度 | 同上 |
| 推理 | BF16 或 FP16 | 省显存，速度更快 |
| 边缘部署 | INT8/INT4 | 极致省显存和功耗 |
| 科学计算 | FP64 | 需要高精度 |
| 老硬件（V100） | FP16 混合精度 | V100 不支持 BF16 |

> A100/H100 支持 BF16 + FP16；V100 只支持 FP16。

## 5. 显存和速度对比

以 7B 参数模型为例：

| 精度 | 参数显存 | 梯度显存 | 优化器状态（AdamW） | 总计（约） |
|---|---|---|---|---|
| FP32 | 28 GB | 28 GB | 56 GB | ~112 GB |
| BF16 混合 | 14 GB（BF16）+ 28 GB（FP32 master） | 14 GB（BF16） | 56 GB（FP32） | ~112 GB |
| BF16 纯低精度 | 14 GB | 14 GB | 28 GB | ~56 GB |
| INT8 | 7 GB | - | - | ~7 GB（推理） |

> 混合精度的"省显存"主要体现在激活值（activation）用 BF16 存储，参数更新仍然需要 FP32。

### 计算速度

- Tensor Core 支持：FP16 和 BF16 都能用 Tensor Core 加速
- A100 BF16 算力：312 TFLOPS（和 FP16 一样）
- H100 BF16 算力：990 TFLOPS

## 6. 面试高频追问

### Q1: BF16 精度只有 2 位有效数字，为什么还能训练？

神经网络训练本质上是一个随机优化过程，梯度和参数更新本身就是带噪声的。2 位有效数字足以区分"大幅更新"和"微调"。实验表明 BF16 和 FP32 的训练曲线几乎一致。

### Q2: 什么情况下必须用 FP32？

- BatchNorm 的均值/方差统计（小 batch 时精度敏感）
- 注意力分数的 softmax（数值稳定性要求高）
- 损失函数的计算
- 参数更新的最终步骤

这些在混合精度框架中由 autocast 自动处理。

### Q3: FP8 是什么？

H100 引入了 FP8（E4M3 和 E5M2 两种格式），进一步将显存需求减半：
- E4M3（4位指数 + 3位尾数）：用于前向
- E5M2（5位指数 + 2位尾数）：用于反向
- 目前主要用于推理和部分训练场景

### Q4: 混合精度会影响模型性能吗？

几乎不会。大量实验表明 BF16 混合精度训练的最终精度与 FP32 训练持平或略好（因为有正则化效果）。FP16 混合精度在某些场景可能略差，但差距在 0.1% 以内。
