### 讲解

这一页给出了本讲的 **大纲与学习目标**（Outline and goals）。核心目标只有一句话：**Make CUDA and GPUs less magic**——让 CUDA 和 GPU 不再是"黑魔法"。

很多深度学习从业者使用 PyTorch 时，对底层的 GPU 运行机制一无所知。代码写完调用 `.cuda()` 就扔到 GPU 上跑，至于为什么有的操作快、有的操作慢，完全是"玄学"。本讲的目标就是打破这种"黑箱"思维，建立对 GPU 性能的直觉。

具体来说，本讲要回答两个关键问题：

**第一，理解 GPU 什么时候会变慢（Understand when GPUs get slow）。** 幻灯片左下角展示了一张来自 Horace He 博客（thonking.ai）的图表，横轴是矩阵维度，纵轴是计算吞吐量（TF/s）。图上标注了几个关键概念：**Tiling**（分块）、**Compute Intensity**（计算强度）、**Wave Quantization**（波量化）。这些都是影响矩阵乘法在 GPU 上实际性能的关键因素。比如，当矩阵维度从小到大变化时，性能并非线性增长，而是呈现出阶梯状的跳跃——这就是 wave quantization 效应。理解这些概念，才能解释为什么某些维度的矩阵乘法出奇地慢。

**第二，理解如何设计快速的算法（Understand how to make fast algorithms）。** 右侧展示了 **FlashAttention**（Dao et al.）的核心思想：通过将数据分块拷贝到 SRAM（片上快速存储）中进行计算，避免反复访问 HBM（高带宽全局显存），从而大幅减少内存访问开销。右上角的柱状图对比了 GPT-2 上标准 PyTorch attention 实现与 FlashAttention 的耗时——FlashAttention 将 Matmul、Softmax、Mask、Dropout 等操作融合为一个 kernel（Fused Kernel），总时间从约 15ms 降低到不足 5ms，加速超过 3 倍。

这两个例子揭示了本讲的核心思想：**GPU 性能优化不仅仅是"算得更快"，更关键的是"搬数据搬得更少"。**

### 讲解

在深入 GPU 架构之前，Tatsu 首先回答一个根本性问题：**为什么计算量（compute）如此重要？**

答案来自 Kaplan et al. 2020 年发表的里程碑论文 *"Neural Scaling Laws"*。图中展示的是语言模型的 **验证集损失（Validation Loss）** 与 **计算量（Compute，单位为 PetaFLOP/s-days）** 之间的关系，不同颜色的曲线代表不同参数量的模型（从 $`10^5`$ 到 $`10^{11}`$）。

核心发现是：**不论模型大小如何，验证集损失与计算量之间存在平滑的幂律关系（power law）**，可以用公式 $`L = 2.57 \cdot C^{-0.048}`$ 来拟合。也就是说，只要你投入更多的计算量——更大的模型、更多的数据、更长的训练时间——模型的性能就会可预测地改善。图中的虚线就是这条拟合曲线，所有不同规模的实验点都紧密分布在它附近。

这个发现的意义深远：它意味着**更快的硬件、更好的硬件利用率、更优的并行化方案，就能直接转化为模型性能的提升**。这不是一个模糊的"可能有帮助"的关系，而是一个精确的、可预测的数量关系。幻灯片底部的总结一针见血：更快的硬件（faster hardware）、更好的利用率（better utilization）、改进的并行化（improved parallelization），单独任何一项都足以推动进步——至少目前是这样（for now）。

这就为整堂课定下了基调：**理解和优化 GPU，就是在直接推动语言模型能力的边界。**

### 讲解

既然计算量如此重要，那**计算能力是怎样增长的？** 这一页回顾了计算硬件的历史演进。

图中展示了著名的 **42 年处理器数据趋势图**（来源：Hennessy & Patterson 的图灵奖演讲，叠加了 Karl Rupp 等人收集的数据）。图中有多条趋势线：

- **晶体管数量（Transistors，橙色）**：持续按照摩尔定律指数增长，从 1970 年代的几千个到 2020 年代的数十亿个。
- **单线程性能（Single-Thread Performance，蓝色）**：在 1980-2005 年间快速增长，但之后增速急剧放缓。
- **频率（Frequency，绿色）**：从 1970 年代的 MHz 级别增长到 2000 年代初的 ~3GHz，然后基本停滞。
- **功耗（Typical Power，红色）**：同样在 2005 年前后触顶，被限制在约 100-200W。
- **逻辑核心数（Number of Logical Cores，黑色）**：从 2005 年之后开始显著增长。

图中标注了一个关键的历史转折点：**Dennard Scaling 的终结**。**Dennard Scaling**（登纳德缩放）是一个与摩尔定律互补的定律：当晶体管尺寸缩小时，电压也同比例降低，因此功耗密度保持不变——这意味着你可以在同样面积、同样功耗下放更多更快的晶体管。从 1980 年到 2000 年代初，Dennard Scaling 使得处理器频率年年攀升，"免费午餐"（H. Sutter 语）源源不断。

但到了 2005 年左右，由于量子隧穿效应和漏电流等物理极限，Dennard Scaling 失效了。频率无法再提升，功耗撞上天花板。这意味着**单纯靠提高时钟频率来获得更快计算的路走不通了**。

底部的总结提出了关键问题：传统的 Dennard Scaling 已经失效，那我们**如何满足 LLM 对计算量的无尽渴求？** 答案在下一页——并行化。

## Slide 7

![Slide 7](面试准备/课程记录/CS336/Lecture_05/images/page_7.png)

### 讲解

答案是：**并行扩展（parallel scaling）仍在继续**，而 GPU 正是并行扩展的核心载体。

这一页引用了 NVIDIA 首席科学家 **Bill Dally** 在 HotChips 大会上的主题演讲数据。右侧图表展示了**单芯片推理性能（Single-Chip Inference Performance）** 在过去 10 年间的增长趋势，纵轴单位为 Int8 TOPS（每秒万亿次 Int8 操作）。从 2012 年的 K20X（3.94 TOPS）到 2023 年的 H100（4000 TOPS，FP8 Transformer Engine），性能增长超过 **1000 倍**。

这 1000 倍的增长来自多个维度的叠加：

- **数值表示（Number Representation）**：从 FP32 → FP16 → Int8 → BF16/TF32 → FP8，降低数值精度可以在同等硬件下执行更多操作。这一项贡献约 **~16x**。
- **专用指令（Complex Instructions）**：引入 DP4（点积）、HMMA（半精度矩阵乘加）、IMMA（整数矩阵乘加）等专用矩阵运算指令，贡献约 **~12.5x**。
- **制程进步（Process）**：从 28nm → 16nm → 7nm → 5nm，晶体管密度的提升带来约 **~2.5x** 的增益。
- **稀疏性（Sparsity）**：A100 引入的结构化稀疏性支持，额外贡献约 **~2x**。
- **模型效率**：模型架构本身的改进也在贡献，综合增益超过 **1000x**。

幻灯片底部用一句斜体的话作为总结：***There is no LLM scaling without GPU scaling***（没有 GPU 的扩展，就没有 LLM 的扩展）。这句话精炼地概括了硬件与算法之间的共生关系：LLM 能力的飞速进步，在底层是由 GPU 算力的指数级增长所支撑的。

### 讲解

接下来进入 GPU 架构的核心内容。首先要回答一个基本问题：**GPU 和 CPU 到底有什么不同？**

幻灯片中央展示了经典的 CPU vs GPU 架构对比图。左侧是 **CPU**：一个大的控制单元（Control），少量的 ALU（算术逻辑单元，图中只有 4 个），大面积的缓存（Cache），以及 DRAM。右侧是 **GPU**：控制单元很小（图中只有几个小的橙色方块），但拥有**大量密集排列的 ALU**（图中绿色方块占据了芯片的绝大部分面积），缓存很小，然后连接 DRAM。

这个对比揭示了两者截然不同的设计哲学：

**CPU 优化的是延迟（latency）**——每个线程尽可能快地完成。为此，CPU 投入大量晶体管用于复杂的控制逻辑（分支预测、乱序执行、推测执行等）和多级缓存系统。这些设计让单个线程的执行速度极快，但能同时运行的线程很少（通常几个到几十个）。打个比方，CPU 就像一个拥有几位顶级厨师的米其林餐厅——每道菜精雕细琢，但同时只能服务少量客人。

**GPU 优化的是吞吐量（throughput）**——同时处理尽可能多的数据。GPU 用大量微小的 ALU 替代了复杂的控制逻辑和大缓存。单个 ALU 很简单，执行速度也不算快，但数量极多（现代 GPU 有数千甚至上万个），可以同时处理海量数据。GPU 就像一个巨大的快餐连锁店——每个工位做的事很简单，但几百个工位同时工作，总产出惊人。

右上角的对比图进一步展示了这个区别：**CPU（低延迟处理器）** 一次运行少量线程（T1-T4），每个线程很快完成但会有大量等待时间；**GPU（高吞吐处理器）** 同时运行大量线程，当一个线程在等待数据时，GPU 立即切换到另一个就绪的线程执行。这种策略叫做 **延迟隐藏（latency hiding）**——虽然每个线程可能经常需要等待（比如等内存数据），但因为有足够多的"候补"线程，计算单元永远不会闲着。

底部还强调了 GPU 的两个重要特点：**很少的分支支持**（因为 SIMT 模型要求所有线程执行相同指令，分支会导致"分支分歧"问题）；**控制逻辑和缓存都很少**（把芯片面积让给了更多的计算单元）。

### 讲解

现在我们进一步"解剖" GPU 的内部结构，首先是**执行单元（execution units）**。

GPU 的计算核心由两级层次构成：

**SM（Streaming Multiprocessor，流式多处理器）** 是 GPU 的基本"工厂单元"。每个 SM 是一个相对独立的处理单元，拥有自己的指令缓存、Warp 调度器、寄存器文件、共享内存/L1 缓存、Tensor Core 等。SM 负责执行一个或多个 **Block（线程块）**——这是 CUDA 编程模型中的工作分配单元。可以把 SM 想象为一个独立的小型工厂，接收任务（Block）并用内部的工人（SP）来完成。

幻灯片左侧展示了一个 SM 的详细内部结构图（GA100 架构）。可以看到它被分为 4 个子区域（对应 4 个 Warp 调度器），每个子区域包含：
- **Warp Scheduler**（Warp 调度器）和 **Dispatch Unit**（分发单元）
- **Register File**（寄存器文件，16,384 × 32-bit）
- 多个 **INT32、FP32、FP64 计算单元**
- **Tensor Core**（用于矩阵乘法加速）
- **LD/ST 单元**（加载/存储单元）和 **SFU**（特殊函数单元）

底部还有 **192KB L1 Data Cache / Shared Memory** 和 **Tex**（纹理单元）。

**SP（Streaming Processor，流式处理器）** 也叫 **CUDA Core**，是 SM 内部的最小计算单元。每个 SP 可以执行一个线程。一个 SM 包含大量 SP——在 GA100 中，每个 SM 有数百个 SP（如 64 个 FP32 单元 × 4 个子区域 = 256 个 FP32 CUDA Core）。

右侧展示了 **GA100 完整 GPU 的芯片布局**，包含 **128 个 SM**。可以看到 SM 以阵列形式排列，占据了芯片的大部分面积，中间是两大块 **L2 缓存**（蓝色区域），周围是 **HBM2(e) PHY**（内存接口）和 **Memory Controller**。

这种层级结构的设计非常优雅：GPU 通过堆叠更多 SM 来增加并行度。下一代 GPU 想要更快？加更多 SM 就行（当然还有其他改进）。这就是为什么 GPU 的计算能力能够持续按 Bill Dally 展示的曲线增长。

### 讲解

理解了执行单元之后，我们来看同样重要的 **内存层次（memory hierarchy）**。幻灯片的标题就给出了最关键的设计原则：**内存离 SM 越近，速度越快（The closer the memory to the SM, the faster it is）**。

GPU 的内存分为四个层级，从快到慢、从小到大：

**1. Shared Memory / L1 Cache（共享内存/L1 缓存）**：物理上位于 **SM 内部**，是最快的片上存储。在 GA100 中，每个 SM 有 192KB 的 L1/共享内存（可以在 L1 cache 和 shared memory 之间灵活分配）。访问延迟约 **23 个时钟周期**（load）/ **19 个周期**（store）。共享内存是程序员可以显式管理的——你可以决定把哪些数据放进去，这是 CUDA 优化的核心技巧之一。

**2. L1 Cache**：同样在 SM 内部，与共享内存共享同一块 SRAM，但由硬件自动管理。访问延迟约 **33 个周期**。

**3. L2 Cache**：位于 **芯片上（on die）**，但在 SM 之外。所有 SM 共享一个大的 L2 缓存（GA100 有 **48MB** 的 L2）。访问延迟约 **200 个周期**——比 L1/共享内存慢了约 6-8 倍。L2 缓存在幻灯片右侧的芯片照片中清晰可见（蓝色的水平条带）。

**4. Global Memory（全局显存，即 HBM/DRAM）**：这些是 GPU 芯片旁边独立的内存芯片（在幻灯片左下角的显卡拆解图中可以看到，标记为 "Video Memory (VRAM)"）。GA100 使用 **HBM2(e)**（高带宽内存），容量可达 **40-80GB**，但访问延迟高达 **290 个时钟周期**——比共享内存慢了 12-15 倍。

左侧的表格（TABLE IV）精确总结了各级内存的访问延迟（CPI，每次访问的时钟周期数）：

| 内存类型 | 延迟（周期） |
|---------|------------|
| Global memory | 290 |
| L2 cache | 200 |
| L1 cache | 33 |
| Shared Memory | 23/19 |

底部的关键结论是：**SRAM（用于共享内存和缓存）比 DRAM（用于全局显存）贵约 100 倍，但快约 8 倍**。这就是为什么 GPU 不能只用 SRAM——太贵了，装不了多少数据；但也不能只用 DRAM——太慢了，计算单元会饿死。这种经济与物理的权衡，决定了 GPU 必须采用分层的内存结构，而高效的 GPU 编程就是要尽量让数据在"快的层"上被反复使用，减少对"慢的层"的访问。这是 FlashAttention 等算法优化的核心思想。

## Slide 11

![Slide 11](面试准备/课程记录/CS336/Lecture_05/images/page_11.png)

### 讲解

了解了硬件结构之后，我们来看 GPU 的 **执行模型（execution model）**——即代码是如何在这些硬件上运行的。CUDA 的执行模型有 **三个关键角色**：

**1. Thread（线程）**：线程是最小的执行单位。GPU 上所有线程执行**相同的指令，但处理不同的数据**——这就是 **SIMT（Single Instruction, Multiple Threads）** 模型。SIMT 可以理解为 SIMD（单指令多数据）的 GPU 版本，但提供了更灵活的编程模型。打个比方：一个工厂流水线上，每个工人做完全相同的动作（同一条指令），但每人手里拿的零件不同（不同的数据）。

**2. Block（线程块）**：Block 是一组线程的集合，是**任务分配的基本单位**。一个 Block 被整体分配到一个 SM 上执行，Block 内的所有线程共享该 SM 的共享内存。幻灯片上方的示意图展示了一个 CUDA 程序由多个 Block 组成（Block 0, Block 1, ... Block 4095），它们被分配（assign）到不同的 SM 上。在本例中，每个 Block 包含 256 个线程。Block 内的线程可以通过共享内存高效通信，但不同 Block 之间**不能直接通信**（必须通过全局显存）。

**3. Warp（线程束）**：Warp 是 GPU 实际调度和执行的基本粒度。**每个 Warp 包含 32 个连续编号的线程**，它们在硬件层面上真正地"锁步执行"——同一时刻执行完全相同的指令。一个包含 256 个线程的 Block 会被划分为 8 个 Warp（256 ÷ 32 = 8）。

幻灯片右侧展示了 SM 内部的 **4 个 Warp 调度器**，每个调度器管理一组 Warp，并将就绪的 Warp 分配给计算单元（INT32、FP32 等）执行。当一个 Warp 在等待内存数据时，调度器会立即切换到另一个就绪的 Warp——这就是前面提到的**延迟隐藏**的机制。因为 Warp 切换几乎零开销（所有 Warp 的状态都常驻在寄存器文件中），GPU 可以极其高效地在不同任务之间切换。

理解 Thread、Block、Warp 的层级关系对于 CUDA 编程至关重要：你在写代码时以 Thread 为单位思考逻辑，以 Block 为单位组织并行任务，而硬件以 Warp 为单位实际调度执行。

## Slide 12

![Slide 12](面试准备/课程记录/CS336/Lecture_05/images/page_12.png)

### 讲解

与执行模型对应的是 GPU 的 **内存模型（memory model）**——不同层级的执行单位可以访问不同层级的内存。

幻灯片右侧的层次结构图清晰展示了这个对应关系：

**线程级别（Thread）**：每个线程拥有自己的 **寄存器（Registers）**，这是最快的存储，但容量极小。寄存器是线程私有的，其他线程无法访问。对于计算密集型操作，将中间结果保持在寄存器中是最高效的做法。

**Block 级别**：同一个 Block 内的所有线程共享一块 **Shared Memory（共享内存）**。如图所示，Block (0,0) 中的 Thread (0,0) 和 Thread (1,0) 都可以访问同一块 Shared Memory。共享内存速度极快（接近寄存器），容量比寄存器大但仍然有限（GA100 中每 SM 最多约 164KB）。共享内存是 Block 内线程间通信的主要渠道。

**跨 Block 级别**：如果不同 Block 之间需要交换数据，就**必须通过 Global Memory（全局显存）**。如图中所示，Block (0,0) 和 Block (0,1) 都连接到底部的 Global Memory 和 Constant Memory。全局显存对所有 Block 和所有线程可见，但访问速度最慢（前面提到的 290 个周期）。

左侧列出了设备端代码（device code）可以执行的内存操作：
- 读写 per-thread 的 **registers**（寄存器）
- 读写 per-thread 的 **local memory**（本地内存，实际在全局显存中，用于寄存器溢出）
- 读写 per-block 的 **shared memory**（共享内存）
- 读写 per-grid 的 **global memory**（全局显存）
- 只读 per-grid 的 **constant memory**（常量内存）

Host（CPU）端代码只能读写 global memory 和 constant memory，通过 PCIe 或 NVLink 传输数据到 GPU。

**关键 takeaway**：跨 Block 的通信代价极高，因为必须经过全局显存。这意味着好的 CUDA 算法设计要尽量让每个 Block 内部的计算自给自足——把需要的数据先搬到共享内存，在共享内存中反复使用，最后再写回全局显存。这就是 **tiling（分块）** 策略的核心思想。

## Slide 13

![Slide 13](面试准备/课程记录/CS336/Lecture_05/images/page_13.png)

### 讲解

这一页是一个"插播"（side thread），简要对比了 **TPU（Tensor Processing Unit）** 与 GPU 的架构异同。

核心观点是：**GPU、TPU 以及其他 AI 加速器，在高层架构上其实非常相似**。它们都遵循同一个设计模式——**Core structure = 轻量级控制 + 快速（大型）矩阵乘法单元 + 快速片上内存**。

幻灯片展示了 TPU **TensorCore**（注意这里的 TensorCore 是 TPU 的计算核心的名字，不要与 NVIDIA GPU 中的 Tensor Core 混淆）的抽象架构图：

- **Scalar Unit（标量单元）**：类似 CPU 的角色，负责指令调度和控制流，将指令分发给 VPU 和 MXU。它连接着 **Smem**（标量内存）。
- **VPU（Vector Processing Unit，向量处理单元）**：执行逐元素运算（如激活函数），并负责将数据从内存加载到 MXU。它有自己的 **Vmem**（向量内存）。
- **MXU（Matrix Multiply Unit，矩阵乘法单元）**：这是 TPU 的核心计算引擎，执行矩阵乘法运算。MXU 是芯片 FLOP/s 的主要贡献者。
- **HBM（High Bandwidth Memory）**：存储权重、激活值、优化器状态、新的 batch 数据等。HBM 带宽决定了数据进出计算单元的速度。

与 GPU 的主要区别：
- **GPU 有更多的 SM**，而 **TPU 的 TensorCore 数量更少**（但每个 TensorCore 的矩阵乘法性能与 GPU 的 SM 相当）。
- TPU **没有 Warp 概念**，只有 Block 级别的组织。这简化了编程模型，但在非矩阵乘法操作上灵活性较低。
- 两者的**网络互联方式不同**——TPU 有专门的高速互联拓扑（如 ICI），而 GPU 使用 NVLink/NVSwitch。这些差异将在并行化相关的课程中进一步讨论。

### 讲解

这一页继续 TPU vs GPU 的对比，提供了更详细的**术语映射和数值对比**。

上方的表格将 GPU 和 TPU 的对应组件一一映射：

| GPU 术语 | TPU 术语 | 功能描述 |

| 组件 | H100 | TPU v5p |
|------|------|---------|
| SM / Tensor Core 数量 | 132 | 2 |
| Warp Scheduler / VPU slots | 528 | 8 |
| SMEM / VMEM 容量 | 32MB（总计） | 128MB |
| Registers / Vector Registers | 32MB（总计） | 256KB |
| Tensor Core / MXU 数量 | 528 | 8 |

最显著的区别是：H100 有 **132 个 SM**，每个 SM 相对较小；而 TPU v5p 只有 **2 个 Tensor Core**，但每个都非常"肥大"（128MB VMEM vs GPU 每 SM 仅约 0.2MB SMEM）。这反映了两种不同的设计哲学：GPU 采用"多而小"的并行策略，TPU 采用"少而大"的策略。两者各有优劣——GPU 的灵活性更好，适合各种不同形状的工作负载；TPU 对大矩阵乘法特别高效，但对不规则的小操作可能不太友好。

底部重申了核心结论：**Core structure = 轻量级控制 + 快速（大型）矩阵乘法单元 + 快速内存**。不管是 GPU 还是 TPU，本讲讨论的性能优化原则——理解内存层次、最大化计算强度、利用分块策略——都同样适用。

## Slide 15

![Slide 15](面试准备/课程记录/CS336/Lecture_05/images/page_15.png)

### 讲解

在完成了 GPU 架构的解剖之后，这一页总结了 **GPU 执行模型的三大优势**：

**优势一：容易扩展大规模工作负载（Easily scales up hard workloads）。** GPU 的并行化是通过增加 SM 数量来实现的。如果需要处理更大的矩阵、更多的数据，理论上只需要更多的 SM。从 Kepler 架构到 Ampere 到 Hopper，NVIDIA 每代 GPU 都在增加 SM 数量（加上每个 SM 内部的改进），从而实现计算能力的持续增长。这种"水平扩展"的特性使得 GPU 天然适合处理深度学习中的大规模并行计算。

**优势二：编程模型相对简单（Easy (?) to program due to the SIMT model）。** 注意幻灯片中的问号——"easy"是加了引号的，因为 CUDA 编程对于初学者来说其实并不简单。但相比于直接编写底层并行代码，SIMT 模型确实提供了很大的便利。程序员可以用类似串行代码的方式编写逻辑（每个线程执行相同的函数），只需要用线程 ID 来决定每个线程处理哪部分数据。硬件负责将这些线程自动映射到数千个执行单元上。图中展示了 SIMT 的工作方式：一个 **Instruction Decoder and Warp Scheduler**（指令解码器和 Warp 调度器）将同一条指令分发给多个 CUDA Core，每个 Core 有自己的寄存器，执行相同的操作但处理不同的数据。

**优势三：线程是轻量级的（Threads are 'lightweight' and can be stopped and started）。** 与 CPU 线程不同，GPU 线程的创建、切换和销毁开销极低。这是因为所有线程的执行上下文（主要是寄存器）都常驻在 SM 的寄存器文件中，切换只需要调度器改变 Warp 的活跃状态，不需要保存/恢复上下文。底部的对比图再次展示了 GPU vs CPU 的线程调度差异：GPU 在任何线程等待数据时可以立即切换到其他线程，最大化计算单元的利用率；而 CPU 核心在一个线程等待时往往只能空闲（或切换代价很高）。

这三个优势共同使 GPU 成为深度学习的理想平台：计算密集（大量相同操作）、数据并行（batch 中的每个样本可以独立处理）、可扩展（模型越大越需要并行）。

## Slide 16

![Slide 16](面试准备/课程记录/CS336/Lecture_05/images/page_16.png)

### 讲解

这一页讲了一段有趣的历史：**GPU 最初并不是为通用计算设计的，而是为图形渲染设计的**——"GPU"的全称是 Graphics Processing Unit。那 GPU 是怎么变成今天的 AI 计算利器的呢？

关键转折点是 **可编程着色器（programmable shaders）** 的出现。早期 GPU 只能执行固定管线的图形操作（光栅化、纹理映射等），但从 2001 年左右（NVIDIA GeForce 3 引入 Vertex Shader 和 Pixel Shader）开始，GPU 开始支持用户自定义的着色器程序。研究人员很快意识到，这些可编程着色器本质上就是可以并行执行的通用计算单元。

幻灯片展示了一篇代表性论文：**"Fast Matrix Multiplies using Graphics Hardware"**，作者 E. Scott Larsen 和 David McAllister（北卡罗来纳大学教堂山分校）。这篇论文展示了研究人员如何"黑入"图形着色器来执行矩阵乘法——一个与图形渲染完全无关的计算任务。右侧的 Implementation 部分甚至提到了一些颇具时代感的 hack：比如"降低显示器刷新率可以提高约 10% 的性能"、"将 4 个数字打包到 RGBA 四通道中"等等。

这段历史说明了两件事：第一，**矩阵乘法一直是 GPU 加速计算的核心用例**，从最早的学术 hack 到今天 NVIDIA 专门设计 Tensor Core 来加速它，矩阵乘法始终是 GPU 存在意义的核心；第二，**GPU 通用计算（GPGPU）** 的需求早在 CUDA（2007 年发布）之前就已存在，CUDA 的诞生是 NVIDIA 对这种需求的正式回应——提供一个比黑入 OpenGL/DirectX 着色器更友好的编程接口。

## Slide 17

![Slide 17](面试准备/课程记录/CS336/Lecture_05/images/page_17.png)

### 讲解

从学术 hack 到今天，GPU 矩阵乘法的性能进化可以用"翻天覆地"来形容。这一页的核心主题是 **Tensor Core**——NVIDIA 从 Volta 架构（V100，2017 年）开始引入的**专用矩阵乘法硬件电路**。

幻灯片中的图表标题是 **"Matmul vs. non-matmul FLOPS across GPUs"**，横轴是不同代 GPU（从 K80 到 H100），纵轴是 TFLOP/s（对数坐标）。两条线分别代表：
- **橙色线（matmul）**：使用 Tensor Core 加速的矩阵乘法性能
- **蓝色线（non-matmul）**：普通浮点运算性能（加法、乘法、非线性函数等）

关键观察：

在 **Tensor Core 出现之前**（K80、M80、P100），matmul 和 non-matmul 的性能基本一致，因为矩阵乘法本质上只是大量的乘加操作，用的是同一套计算单元。

从 **V100 开始**（Volta 架构引入 Tensor Core），matmul 性能开始与 non-matmul 拉开巨大差距。到 H100，**matmul 性能约为 non-matmul 的 10 倍以上**（约 1000 TFLOP/s vs 约 67 TFLOP/s）。

**Tensor Core 为什么这么快？** 因为它是一个专门为小矩阵乘法设计的硬件电路（不是通用的 ALU）。每个 Tensor Core 可以在单个时钟周期内完成一个小矩阵的乘加运算（如 4×4 的 FP16 矩阵乘法并累加到 FP32），而用普通 CUDA Core 做同样的运算需要许多个周期。这就像用计算器算加法（通用 ALU）和用乘法表直接查答案（Tensor Core）的区别。

这带来了一个深刻的实际启示：**在现代 GPU 上，尽可能将计算表达为矩阵乘法（matmul）是最重要的优化策略之一**。如果你的算法可以被重构为矩阵乘法，它就能利用 Tensor Core 获得 10 倍以上的加速；如果不能，你就只能用相对慢得多的通用计算路径。这也是为什么 Transformer 架构（其核心计算几乎全是矩阵乘法）在现代 GPU 上特别高效——它完美契合了硬件的能力分布。

## Slide 18

![Slide 18](面试准备/课程记录/CS336/Lecture_05/images/page_18.png)

### 讲解

这一页揭示了现代 GPU 编程中最关键的矛盾：**计算能力的增长速度远快于内存带宽的增长速度——这就是"内存墙"（Memory Wall）问题**。

图表标题是 **"Scaling of Peak hardware FLOPS, and Memory/Interconnect Bandwidth"**，横轴是年份（1996-2023），纵轴是归一化增长倍数（对数坐标，以某个基准年为 1）。三条趋势线清楚地展示了不同硬件资源的增长速率差异：

- **HW FLOPS（硬件计算能力，黑色）**：20 年增长了 **60,000 倍**，即每 2 年约增长 **3.0 倍**。
- **DRAM BW（内存带宽，绿色）**：20 年增长了 **100 倍**，即每 2 年约增长 **1.6 倍**。
- **Interconnect BW（互联带宽，蓝色）**：20 年增长了 **30 倍**，即每 2 年约增长 **1.4 倍**。

也就是说，**计算能力的增长速度是内存带宽增长速度的约 600 倍**（60,000x vs 100x）。这意味着什么？意味着随着时间推移，**给计算单元"喂"数据变得越来越难**。计算单元越来越快，但数据从内存搬到计算单元的管道并没有按同等比例变宽。

这就像一个工厂不断雇更多的工人（增加 FLOPS），但运输原材料的卡车数量（内存带宽）增加得远没有那么快。工人们越来越多时间在等待原材料，而不是在加工产品。

图中标注了具体的硬件里程碑：内存技术从 GDDR3 → GDDR4 → GDDR5 → HBM → HBM2 → HBM2E，互联从 PCIe 1.0a → PCIe 5.0 → NVLink 1.0-4.0，计算芯片从 Pentium II Xeon → GTX 580 → K40 → A100 → H100。

底部的总结一针见血：**FLOPs scale faster than memory – it's hard to keep our compute units fed with data!**（FLOP 增长快于内存——让计算单元有活干越来越难了！）

这是后续所有性能分析和优化的基础：大多数深度学习操作实际上是**内存受限（memory-bound）** 而非**计算受限（compute-bound）** 的。理解这一点，就理解了为什么 FlashAttention（通过减少内存访问来加速）比简单地购买更多 FLOPS 要有效得多。

### 讲解

为了让 GPU 跑得更快并突破内存墙的限制，我们通常有以下 6 种常用的优化技巧（Tricks）：

1. **Control divergence（控制分歧）**：这主要是执行效率问题，而非内存瓶颈。
2. **Low precision computation（低精度计算）**：减少每次传输的数据量。
3. **Operator fusion（算子融合）**：减少内核启动和中间结果的读写。
4. **Recomputation（重计算）**：用计算时间换取内存读取时间。
5. **Coalescing memory（合并内存访问）**：最大化内存带宽利用率。
6. **Tiling（分块计算）**：终极武器，通过共享内存重用数据。

## Slide 23

![Slide 23](面试准备/课程记录/CS336/Lecture_05/images/page_23.png)

### 讲解

**技巧 1：控制分歧（Control Divergence）**

需要澄清的是，这并不是一个内存问题，而是由 GPU 的 **SIMT（单指令多线程）**执行模型引起的。在 GPU 中，32个线程组成一个 Warp 作为一个整体一起执行同一条指令。

如果代码中包含条件分支（例如 `if-else` 语句），并且同一个 Warp 中的不同线程进入了不同的分支，就会发生**控制分歧**。此时，GPU 无法同时执行两条不同的指令路径。它的处理方式是：先让一部分线程执行 `if` 分支，同时屏蔽掉本该执行 `else` 分支的线程；等 `if` 分支执行完后，再反过来执行 `else` 分支，屏蔽之前的线程。

这意味着，一旦发生分歧，Warp 的执行时间是两个分支时间的总和，而不是最大值，大量线程在等待中处于空闲（idle）状态，导致计算资源的严重浪费。因此，在 CUDA 编程中，应尽量避免在 Warp 内部出现复杂的分支逻辑。

## Slide 24

![Slide 24](面试准备/课程记录/CS336/Lecture_05/images/page_24.png)

### 讲解

**技巧 1：低精度计算（Low Precision Computation）**

对于初学者来说，提升 GPU 性能最简单粗暴的方法之一就是：**降低数据精度**。

**什么是精度？**
在计算机中，所有的数字都是用二进制的“位”（bit，也就是 0 或 1）来表示的。
- 传统的深度学习主要使用**单精度浮点数（Float32，简称 FP32）**。它用 32 个 bit（相当于 4 个字节，Byte）来精确表示一个小数。
- 后来，大家发现神经网络其实不需要那么精确的小数，于是引入了**半精度（Float16 或 Bfloat16）**，它只用 16 个 bit（2 个字节）来表示一个数。
- 现在的前沿甚至在使用 **Float8（FP8）**，只用 8 个 bit（1 个字节）来表示一个数。

**为什么降低精度能让程序变快？**
我们可以打个比方：GPU 的“全局内存”是一个巨大的仓库，“计算单元（SM）”是加工厂，而“内存带宽”是连接两者的卡车车队。
如果你用 Float32，每个数字就像是一个大箱子（占 4 个字节）。一辆卡车一次只能拉 10 个大箱子。
如果你换成 Float16，每个数字变成了小箱子（占 2 个字节）。同样的卡车，一次就能拉 20 个小箱子！换成 Float8，一次能拉 40 个！

**核心思想**：“如果你要移动的比特（bits）变少了，那么移动它们花费的时间也就变少了。” 
在现代深度学习中，绝大多数的瓶颈都在于“卡车运货太慢”（内存带宽不足），而不是“工厂加工太慢”（计算能力不足）。通过使用低精度，我们在相同的时间内给工厂输送了成倍的数据，工厂就不会因为等原材料而停工，整体速度自然就翻倍了。

## Slide 25

![Slide 25](面试准备/课程记录/CS336/Lecture_05/images/page_25.png)

### 讲解

**算术强度（Arithmetic Intensity）是如何被提升的？**

上一页我们用了卡车的比喻，这一页我们用一个极其重要的专业指标来量化它：**算术强度**。
它的公式非常简单：**算术强度 = 计算次数 (FLOPs) / 传输的字节数 (Bytes)**。
它代表了“工厂每运进来 1 字节的数据，能做多少次计算”。这个数值越大，说明工厂的效率越高，越不容易被卡车运输速度拖后腿。

让我们看一个具体的例子：对一个数组做 **ReLU 激活函数**操作（把所有负数变成 0，正数保持不变，公式是 $`x = \max(0, x)`$）。

- **在使用 Float32 时**：
  为了处理一个数字，GPU 首先要把它从内存读进来（读 4 字节），然后做一个大小比较（算 1 次计算，即 1 FLOP），最后把结果写回内存（写 4 字节）。
  总共移动了 8 字节，做了 1 次计算。
  算术强度 = **1 / 8 = 0.125 FLOPs/byte**。这是一个极低的数字，意味着 GPU 大部分时间都在等数据搬运。

- **在使用 Float16 时**：
  同样是读进来（这次只有 2 字节），做一次比较（1 FLOP），写回去（2 字节）。
  总共移动了 4 字节，做了 1 次计算。
  算术强度 = **1 / 4 = 0.25 FLOPs/byte**。

你看！仅仅是把数据类型从 FP32 换成 FP16，我们就把**算术强度翻倍**了。这意味着在内存读取速度不变的情况下，GPU 的计算单元能获得多一倍的数据去处理，等待时间大幅减少。

## Slide 26

![Slide 26](面试准备/课程记录/CS336/Lecture_05/images/page_26.png)

### 讲解

**硬件层面的终极偏爱：Tensor Cores**

低精度的另一个巨大优势不仅在于“运货变快”，更在于 GPU 内部有一套专门为低精度准备的“超级流水线”。

在普通的 GPU 运算中，我们使用的是普通的 **CUDA Cores**，它们是通用的处理器，什么都能算。
但是从 Volta 架构（如 V100 显卡）开始，NVIDIA 在 GPU 里塞进了一种叫做 **Tensor Cores（张量核心）** 的专用小怪物硬件。

**什么是 Tensor Cores？**
你可以把它理解为一个专门只做一件事的机器：**极速进行小矩阵乘法并相加（MMA，Matrix Multiply Accumulate）**。
普通的 CUDA Core 计算矩阵乘法是一步一步做乘加的，而 Tensor Core 可以在“一个时钟周期”内，一口气把一个 $`4 \times 4`$ 的小矩阵直接乘完并加到结果上。

**这和低精度有什么关系？**
关键在于，Tensor Cores 被设计为**极度偏爱低精度（混合精度）**！
它被设定为：接收 FP16（或 FP8）格式的输入数据，然后在内部用高精度（FP32）进行累加，以保证不丢失精度。
如果你的代码使用了 FP16 并涉及到矩阵乘法，GPU 就会自动激活 Tensor Cores。此时，你的计算吞吐量会瞬间爆发出普通 CUDA Core **十倍以上**的威力！

所以，使用低精度是一箭双雕：既减轻了内存搬运的压力，又激活了 GPU 内部最强大的计算引擎。

## Slide 27

![Slide 27](面试准备/课程记录/CS336/Lecture_05/images/page_27.png)

### 讲解

**向极致压榨：FP8 的挑战与魔法（Scaling Factors）**

既然 FP16 这么好用，我们能不能继续压缩，用 **FP8（8个bit，1个字节）** 呢？
当前的 Blackwell 等最新架构正是这么做的，但这里遇到了一个巨大的工程难题。

**FP8 的困境是什么？**
8 个 bit 只有 256 种组合。也就是说，你只能表示 256 个不同的数字。
在浮点数设计中，这 8 个 bit 要分给“指数（Exponent，决定数值的范围有多大）”和“尾数（Mantissa，决定数值的精度有多细）”。
- 如果你选 **E4M3** 格式（4位指数，3位尾数）：你的精度还不错，但是能表示的最大数字很小。
- 如果你选 **E5M2** 格式（5位指数，2位尾数）：你的范围变大了，但是数字之间的间隔非常粗糙（精度差）。
神经网络训练既需要大范围，又需要细精度，这可怎么办？

**魔法解法：共享缩放因子（Shared Scaling Factors）**
NVIDIA 的架构（如 MXFP8 标准）引入了一个绝妙的设计。它们选择使用精度较好的 E4M3，然后额外配一个“放大镜”——**缩放因子（Scaling Factor）**。
你可以把缩放因子理解为一个乘数（比如 $`\times 1000`$）。原本 FP8 只能表示极小的小数，乘上这个缩放因子后，瞬间就变成了大数。
为了节省空间，并不是每个数字都配一个缩放因子。而是**每连续的 32 个数字组成一个小组（Block），这 32 个小弟共享同一个“缩放因子”大哥**。
更夸张的是，这个“缩放因子”大哥本身也被压缩成了一个 8 bit 的数（E8M0 格式）。

这种极其精密且复杂的设计，让大语言模型得以在极小的显存占用下保持高精度的计算。

## Slide 28

![Slide 28](面试准备/课程记录/CS336/Lecture_05/images/page_28.png)

### 讲解

**FP8 带来的工程噩梦：矩阵转置**

虽然 FP8 加缩放因子的方案看起来很完美，但在实际写代码时却引发了意想不到的麻烦，尤其是在做**矩阵转置（Transposes）** 的时候。

**为什么转置变难了？**
在以前的 FP32 时代，矩阵转置非常简单：原本在内存中横着读数据（读行），现在竖着读数据（读列）就行了。
但在 MXFP8 格式下，由于**每 32 个连续的元素共享一个缩放因子**，这 32 个元素在物理内存上必须被绑在一起！
如果矩阵是按行存储的，那么同一行的 32 个元素共享一个因子。
当你要做转置，开始竖着读（按列读取）时，你每读一个元素，它都属于不同的行，也就跟着**不同**的缩放因子！
这意味着你无法轻易地提取出一整列数据并作为一个新的 32 人小组，因为它们各自的缩放因子不匹配，你必须在底层重新解压、重新计算新的缩放因子、再重新打包。

这就要求深度学习框架（比如 PyTorch 的底层编译器）变得非常聪明，对于转置的矩阵，它必须有一套完全独立且复杂的内存管理和量化策略。

### 讲解

**疯狂的未来：MXFP4（4-bit 计算）**

如果 8-bit 还不够小，那么 **4-bit（MXFP4）** 就是目前的终点。

4 个 bit，意味着总共只能表示 $`2^4 = 16`$ 种可能的值！
这听起来简直不可思议：用仅仅 16 个不同的数字，就能拼凑出会写诗、写代码的 ChatGPT 吗？
答案是：能。前提是依然使用上一页提到的**分组缩放因子（Micro-scaling）**技术。
在 MXFP4 标准中，每 16 个 4-bit 元素被捆绑在一起，共享一个额外的缩放因子。

这种极限压缩对“大语言模型（LLM）推理”至关重要。
因为在实际部署模型时（比如让模型回答用户问题），最大的瓶颈是：模型的权重文件太大，不仅要买很多昂贵的 GPU 来装载它们，并且把权重从显存搬运到计算单元的过程（Memory Bound）极其缓慢。
MXFP4 将显存需求和带宽需求直接压缩到了原本（FP16）的四分之一。这是目前让巨大模型能够在少数几张甚至单张消费级显卡上流畅运行的最核心科技。

## Slide 30

![Slide 30](面试准备/课程记录/CS336/Lecture_05/images/page_30.png)

### 讲解

**技巧 2：算子融合（Operator Fusion）**

刚才我们聊了怎么把货物变小（低精度），现在我们要聊聊怎么**优化加工流程**。

我们再回到“仓库（Global Memory 显存）”和“工厂（SM 计算单元）”的比喻。
在深度学习中，通常有一连串的操作。比如：先做加法（操作 A），再做激活函数（操作 B），最后做归一化（操作 C）。

**传统做法（未融合）**是什么样子的？
1. 工厂派卡车去仓库，把材料拉回来。
2. 工厂做加法操作（A）。
3. 做完后，把半成品装上卡车，运回仓库**存起来**。
4. 紧接着，马上又派卡车去仓库，把刚才的半成品再**拉回工厂**！
5. 工厂做激活函数操作（B）。
6. 再运回仓库。
...以此类推。

你能看出来问题在哪里吗？绝大部分时间根本没在加工，全浪费在卡车把半成品运来运去的路上了！由于内存带宽（运货速度）远远跟不上计算能力（加工速度），这种做法导致 GPU 处于极其低效的“内存受限（Memory Bound）”状态。

**算子融合（Operator Fusion）**的核心思想极其直白：
**既然数据都已经拉进工厂了，为什么不一口气把 A、B、C 所有工序全做完，最后只把最终成品运回仓库呢？**
这就避免了中间半成品无意义的来回搬运。

### 讲解

**朴素内核（Naive Kernel）vs 融合内核（Fused Kernel）的对比图**

这张图直观地展示了为什么算子融合能大幅提升速度。图中表示了时间的流逝。

结果就是：完成同样的工作，融合后的总时间（横轴长度）大幅缩短。

**一个反面教材：未融合的 $`y = \sin^2(x) + \cos^2(x)`$**

为了让大家更深刻地理解，我们来看一段看起来非常正常的 PyTorch 代码：

![Slide 34](面试准备/课程记录/CS336/Lecture_05/images/page_34.png)

### 讲解

**技巧 3：重计算（Recomputation / Activation Checkpointing）**

在这个技巧中，我们将遇到一种“反直觉”的优化策略：**为了跑得更快、训得更大，我们宁愿把已经算好的结果扔掉，一会儿再重新算一遍！**

**背景知识**：在训练神经网络时，有一个著名的过程叫“反向传播（Backpropagation）”用于计算梯度更新权重。
麻烦在于，要计算某一层在反向传播时的梯度（图中的绿色方块），你不仅需要反向传回来的误差信号，你**还必须拥有这一层在前向传播（Forward pass）时产生的原始输出结果（即“激活值”，图中的黄色方块）**。

传统的做法很老实：在前向传播时，系统会把每一层产生的黄色方块统统保留在全局内存（显存）里，就像存盘一样，一直等到反向传播时再拿出来用。
但这里有个致命问题：大型语言模型有几十甚至上百层，如果把每一层的中间结果都存在显存里，你的显存瞬间就会被撑爆，这就是为什么你经常遇到 `CUDA Out Of Memory` (OOM) 报错的原因。

### 讲解

**保存激活值不仅费空间，还极度拖慢速度！**

我们来看一个 3 层 Sigmoid 神经网络的极端例子。
如果在前向传播中，我们要老老实实地保存每一层的激活值以便后用，我们需要做什么？

1. 输入 $`x`$。

你会发现，这个过程中发生了大量的内存读写（总计 8 次！）。
这又是一个典型的“算术强度极低”的场景——为了做一点点求导的计算，卡车在仓库和工厂之间跑了无数趟。这不但占满了显存空间，大量的内存读写带宽消耗也严重拖慢了训练速度。

## Slide 36

![Slide 36](面试准备/课程记录/CS336/Lecture_05/images/page_36.png)

### 讲解

**破局之道：丢掉它！等需要的时候再重算一遍！**

这就是 **重计算（Recomputation）** 或称 **激活检查点（Activation Checkpointing）** 的核心魔法。

我们的策略变成了这样：
在前向传播时，算完 $`A1`$，**直接扔掉，根本不往全局内存里存！** $`A2`$ 算完也直接扔掉！只保留最少量的关键节点。
由于我们没有存数据，这就节省出了海量的显存空间（你可以借此把模型的 Batch Size 开得更大）。

**那反向传播需要 $`A1`$ 和 $`A2`$ 的时候怎么办？**
很简单，当梯度传回到那一层时，我们利用输入数据 $`x`$，**现场重新做一次前向传播计算**，把 $`A1`$ 重新算出来！

**初学者最大的疑问：多做了一次计算，不会变得更慢吗？**
这就是理解 GPU 性能最关键的转折点：**在 GPU 上，“做数学计算”的速度，远远远远快于“从全局内存读数据”的速度！**
如果你存下来，读取它需要 100 毫秒；但如果你丢掉它现场重算，计算本身可能只要 10 毫秒。
所以，重计算是用极其廉价的“纯计算资源（Compute）”，去换取极其昂贵和稀缺的“显存容量和内存读写带宽（Memory Bandwidth）”。
结果就是：重计算不但大幅降低了显存占用，在很多情况下，因为免除了大量的读写等待，甚至会让整体训练速度变得更快！

## Slide 37

![Slide 37](面试准备/课程记录/CS336/Lecture_05/images/page_37.png)

### 讲解

**技巧 4：内存合并（Memory Coalescing）与 DRAM 工作原理**

如果说前面的技巧是在教你怎么“少用卡车运货”，那么接下来的技巧就是教你“**如何把每一辆卡车都装得满满当当**”。
这就涉及到 **内存合并访问（Memory Coalescing）**。

为了理解这个概念，我们必须深入了解物理内存（DRAM，也就是全局显存）是怎么工作的。
请把 DRAM 想象成一个巨大的、陈旧的文件柜，里面有无数排抽屉。
当你要找某一份特定文件（读取某个字节）时，由于硬件结构的限制，DRAM 是无法“只抽出薄薄的一页纸”递给你的。
它的工作方式是所谓的 **“突发模式（Burst Mode）”**：只要你请求了某一个位置的数据，内存控制器就会不管三七二十一，**把你请求的那个位置及它周围连续的一大段数据（通常是连续的 32 字节或 128 字节）全部一次性“搬出来”**扔给缓存。

这是一个极其笨重且缓慢的物理过程。因此，我们必须顺应它的脾气。

## Slide 38

![Slide 38](面试准备/课程记录/CS336/Lecture_05/images/page_38.png)

### 讲解

**好与坏：合并访问（Coalesced） vs 非合并访问（Uncoalesced）**

前面提到过，GPU 是以 Warp（32个线程为一组）为单位干活的。当这 32 个线程同时需要向内存请求数据时，会发生两种情况：

**好情况：合并内存访问（Coalesced Memory Access）**
如果这 32 个线程很有默契，线程0要第0个数据，线程1要第1个数据，线程2要第2个数据……它们需要的数据在内存物理地址上是**完全连续挨在一起的**。
这时候硬件会非常开心！因为这 32 个数据恰好落在 DRAM 的一次“突发（Burst）”提取范围内。内存控制器只需要向 DRAM 下达**1 次读取指令**，一车就拉回了所有线程需要的数据。这叫完美的“合并访问”，效率最高。

**坏情况：非合并访问（Uncoalesced Memory Access）**
如果这 32 个线程乱来：线程0要第0个位置，线程1要第100个位置，线程2要第5000个位置……它们需要的数据天女散花般散落在内存的不同角落。
这时候，虽然这 32 个线程同时发起了请求，但因为地址不连续，无法落在同一次 Burst 里。硬件被迫必须向 DRAM 下达 **32 次独立的读取指令**！
这就好比，为了拿 32 张纸，你不得不抽出 32 个不同的笨重抽屉。并且每次抽屉拉出来一堆数据，你只要其中的一页，剩下的全被扔掉了（带宽严重浪费）。
这种非合并访问会导致程序的内存读取速度瞬间暴跌 10 倍甚至几十倍！

## Slide 39

![Slide 39](面试准备/课程记录/CS336/Lecture_05/images/page_39.png)

### 讲解

**一个现实的噩梦：矩阵乘法中的列读取**

明白了合并访问的道理，我们再来看最常见的操作——**矩阵乘法 $`C = A \times B`$** 中隐藏的巨大陷阱。

在计算机的内存中，矩阵是变成一根长长的直线存储的。默认采用**行优先（Row-major）**：就是第一排存完了，紧接着存第二排，再存第三排。
这意味着：**矩阵中同一行的数据，在物理内存中是连续挨着的。但同一列的数据，在物理内存中隔得很远（要跨过一整行的数据）。**

在算矩阵乘法时，我们需要用矩阵 A 的“行”去乘矩阵 B 的“列”。
- **读矩阵 A 的行**：这是非常美好的！线程组顺着行读，地址是连续的，完美实现了**合并读取（Coalesced）**。
- **读矩阵 B 的列**：这就成了噩梦！线程0读第0行第0列，线程1读第1行第0列……它们在内存地址上隔了一万八千里。这就引发了极其严重的**非合并访问（Uncoalesced）**。为了仅仅凑齐一列的数据，GPU 被迫要发起几十次甚至几百次低效的内存事务，导致性能被彻底锁死。

这就是为什么你自己用简单的 for 循环写出来的矩阵乘法，永远比 PyTorch 自带的慢 100 倍的原因之一。

![Slide 40](面试准备/课程记录/CS336/Lecture_05/images/page_40.png)

### 讲解

**除了乱读，还有更惨的：重复读（Tiling 优化的铺垫）**

矩阵乘法的痛点不止在于读取列时的“非合并访问”，还有一个极其严重的问题：**大量重复的无用功**。

在朴素的矩阵乘法中，我们要算出结果矩阵 C 里的每一个格子。
比如，为了算 C 左上角的第一个格子 $`P_{0,0}`$，线程必须把 A 的第一行和 B 的第一列从全局内存（最慢的仓库）里完完整整读一遍。
当你紧接着要算它下方的格子 $`P_{1,0}`$ 时，那个线程又得去读取一遍 A 的第二行，和 **B 的第一列**！
你发现了吗？**B 的第一列被从慢速内存中重复读取了无数次！** 矩阵 A 的同一行也被后续的列重复读取了无数次。

全局内存（Global Memory）的速度本来就慢得像乌龟，这种对同一批数据 $`N \times N`$ 次的重复读取，是对内存带宽极大的浪费。
我们急需一种方法：**能不能把数据读出来一次，就存到一个非常快的地方，让大家共享用完再扔？**
答案就是技巧 5：**Tiling（分块）**。

## Slide 41

![Slide 41](面试准备/课程记录/CS336/Lecture_05/images/page_41.png)

### 讲解

**技巧 5：分块（Tiling）——现代 GPU 优化的灵魂**

为了解决全局内存重复读取和非合并访问的问题，大杀器 **Tiling（分块/切片）** 登场了。

它的核心思想是利用 GPU 内部一块非常神奇且极速的存储区域——**共享内存（Shared Memory，简称 SMEM）**。
共享内存就建在计算单元（SM）的内部，速度快如闪电，并且同一个线程块（Block）里的所有线程都可以自由读取里面的数据（这也是为什么它叫共享内存）。

**Tiling 的运作方式就像搬砖小队的分工合作：**
我们不再让每个线程傻傻地独立去全局内存捞数据了。我们把大矩阵切成一个个小的“方块（Tiles）”。
1. **协作加载（Load）**：一个线程块里的一群线程通力合作，以**合并访问（最高效）**的方式，将矩阵 A 的一个小方块和矩阵 B 的对应小方块，从缓慢的全局内存一次性“搬运”到极速的共享内存中暂存。
2. **高速计算（Compute）**：数据进了共享内存后，所有线程都在这个极速的缓存里拿数据做矩阵乘法。由于大家都能访问共享内存，原来需要被重复读取 $`N`$ 次的数据，现在只在全局内存读了 $`1`$ 次，而在共享内存里被反复重用了 $`N`$ 次！
3. **滑动推进（Slide）**：这一个小块算完了，线程小队再合作把下一个小块搬进共享内存覆盖旧数据，继续算。

这就彻底解决了之前的所有痛点：全局内存读取全部合并了，且没有一丝数据的重复搬运。

## Slide 42

![Slide 42](面试准备/课程记录/CS336/Lecture_05/images/page_42.png)

### 讲解

**算一笔账：Tiling 的数学威力**

让我们用简单的数学证明，为什么 Tiling 能带来巨大的性能提升。

假设我们要算两个 $`N \times N`$ 大小的矩阵相乘。
- **如果不使用 Tiling（朴素算法）**：为了算出结果矩阵的每个点，每个输入矩阵的数据点都会被从全局内存里读取 $`N`$ 次。总的内存访问量是恐怖的 $`O(N^3)`$ 级别。
- **如果使用 Tiling（假设方块大小是 $`T \times T`$）**：在一个小队（线程块）内部，这 $`T \times T`$ 的数据被搬进极速的共享内存后，内部的每个线程都会反复利用它 $`T`$ 次。这意味着，相对外面的全局内存而言，每个数据被读取的次数从 $`N`$ 骤降到了 $`N/T`$。

**结论**：使用大小为 $`T`$ 的分块，可以**将最致命的全局内存访问量直接减少 $`T`$ 倍！**
在 GPU 这个极度受制于内存瓶颈（Memory Bound）的环境中，让卡车少跑 $`T`$ 倍的路，基本上就等同于让整体计算速度直接飙升了 $`T`$ 倍！这就是为什么所有高性能深度学习算子（如 cuBLAS）内部全部都采用了极其复杂的 Tiling 架构。

### 讲解

**既然 Tiling 这么好，为什么写起来这么难？（复杂性 1）**

虽然从上一页看，$`T`$ 越大越好，能节省的内存带宽越多。但现实中，决定一个完美的方块大小（Tile Size）是一门玄学，充满了妥协。

1. **共享内存容量极其有限**：在 A100 GPU 上，一个 SM 内部的共享内存总共只有不到 200 KB。如果你把 Tile 设得太大，根本就塞不进去。不仅如此，如果你的 Tile 占用了所有的共享内存，那么这台机器就无法同时接待其他的线程队伍（降低了 Occupancy 占用率），导致 GPU 计算单元闲置。
2. **让人头疼的整除问题（Divisibility）**：假设你费尽心机算出了最完美的 Tile 大小是 $`128 \times 128`$。结果用户传进来的矩阵大小是 $`1000 \times 1000`$。尴尬的事情发生了：$`1000`$ 除以 $`128`$ 除不尽（有余数 $`104`$）！
这就意味着在矩阵的边缘，你的 Tile 会“超出边界”。为了防止程序读到内存之外的地方导致崩溃（Segfault），你必须在 CUDA 代码里写一大堆啰嗦的 `if-else` 来检查边界条件。而我们前面学过，GPU 最讨厌 `if-else` 分歧！这就导致边缘计算的效率大打折扣。

## Slide 44

![Slide 44](面试准备/课程记录/CS336/Lecture_05/images/page_44.png)

### 讲解

**更底层的痛：内存对齐（Memory Alignment）与填充（Padding）**

还记得 Slide 37 里的抽屉比喻吗？DRAM 的读取是成块的（Burst），比如一次必须拿连续的 128 个字节，而且必须从特定的“对齐边界（比如 0, 128, 256）”开始拿。

如果你精心设计的矩阵一行的长度，刚好跨越了硬件的“对齐边界”，一场悲剧就会发生。
举个例子：你想读取某个方块的第一行数据。但是这行数据在物理内存中的起始位置偏离了对齐边界。导致内存控制器原本只需拉开一个抽屉就能拿到的数据，现在必须**拉开两个相邻的抽屉**才能凑齐！然后下半截又被扔掉。这就造成了巨大的带宽浪费。

**怎么解决？使用填充（Padding）**
为了迎合这个古怪的硬件脾气，系统会在矩阵的右侧或者下方人为地补上一堆**没有任何意义的“0”**。
通过强行增加行的长度，使得矩阵的每一行在物理内存上的开头，都严丝合缝地对齐在 DRAM 的 Burst 边界上。这就是为什么在深度学习里，通常要求矩阵的大小最好是 8、32 或者 128 的倍数——因为这能让底层硬件跑得最舒服。

![Slide 48](面试准备/课程记录/CS336/Lecture_05/images/page_48.png)

### 讲解

**解密 2：波形量化（Wave Quantization）**

为什么在 1792 到 1793 之间，矩阵维度仅增加 1，性能却出现周期性的断崖式下跌？这就是著名的 **Wave Quantization 效应**（也叫尾部效应）。

假设我们使用的 Tile 大小是 $`256 \times 128`$。
- 对于大小为 **1792** 的矩阵：$`1792 / 256 \times 1792 / 128 = 7 \times 14 = 98`$ 个块（Tiles）。
- 对于大小为 **1793** 的矩阵：由于不能整除，必须向上取整，分配 $`8 \times 15 = 120`$ 个块。

重点来了：一台 A100 GPU 通常只有 108 个 SMs。
- 只有 98 个块时，GPU 的 108 个 SM 可以一次性（在一个 Wave 内）同时执行完所有的块，非常高效。
- 当有 120 个块时，108 个 SM 第一波先执行 108 个块，然后**所有 SM 必须等待第一波执行完毕，才能去执行剩下的 12 个块**。这就导致在第二波计算时，有 $`108 - 12 = 96`$ 个 SM 处于完全闲置状态。这种严重的资源浪费直接导致了整体性能的急剧下降。

![Slide 49](面试准备/课程记录/CS336/Lecture_05/images/page_49.png)

### 讲解

**Part 2 总结：如何让 ML 工作负载跑得更快**

要想在 GPU 上实现极致的性能，核心是解决“内存墙”问题。回顾本节课介绍的技巧，本质上可以归为三大流派：

1. **绝对地减少全局内存访问次数**：
   - **内存合并（Coalescing）**：让每次访存效率最大化。
   - **算子融合（Fusion）**：在寄存器级别消化掉中间结果，避免写回全局内存。

2. **将数据移动到更靠近计算单元的内存层级**：
   - **分块（Tiling）**：将全局内存的数据搬运到极速的共享内存（Shared Memory）中循环重用。

3. **用计算或精度来换取内存带宽**：
   - **量化（Quantization/Low Precision）**：减少每次传输的比特数。
   - **重计算（Recomputation）**：宁愿在反向传播时重新计算，也不要把中间结果存到内存里再读出来。

理解并熟练运用这些底层技巧，是将算法落地为高效运行代码的关键。

# Part 3: 综合运用 — 深入理解 FlashAttention

## Slide 50

![Slide 50](面试准备/课程记录/CS336/Lecture_05/images/page_50.png)

### 讲解

这一部分是整个 Lecture 5 的高潮，我们将把前面学到的所有 GPU 优化知识——**内存层次结构、tiling、kernel fusion**——综合运用，来理解一个在现代大语言模型训练中极其重要的优化技术：**FlashAttention**（由 Tri Dao 等人提出）。

**FlashAttention** 的核心问题是：标准的 Transformer attention 计算在 GPU 上为什么这么慢？答案不在于计算量本身，而在于**内存访问**。让我们从数据来理解这一点。

从 GPT-2 上 attention 操作的时间分解柱状图可以清晰地看到：标准 PyTorch 实现需要约 17ms，其中包含了多个独立的操作——两次矩阵乘法（Matmul）、Mask、Softmax、Dropout。每一个操作都是一个独立的 CUDA kernel，意味着中间结果需要反复写入 HBM（高带宽显存）再读回来。而 FlashAttention 将所有这些操作融合成一个单一的 **Fused Kernel**，运行时间仅约 3ms，加速了约 5-6 倍。

性能对比表格揭示了更深层的原因：

| 指标 | Standard Attention | FlashAttention |
|------|-------------------|----------------|
| GFLOPs | 66.6 | 75.2 |
| HBM R/W (GB) | 40.3 | **4.4** |
| Runtime (ms) | 41.7 | **7.3** |

注意一个极其重要的现象：FlashAttention 的 **GFLOPs 反而略高**（75.2 vs 66.6），也就是说它实际做的浮点运算比标准实现还多！但它的运行时间却从 41.7ms 降到了 7.3ms——快了近 **6 倍**。这怎么可能？秘密在于 HBM 读写量：从 40.3 GB 骤降到 4.4 GB，减少了约 **9 倍**。这完美印证了我们之前学到的核心观点：**attention 是一个内存受限（memory-bound）的操作，瓶颈不在计算而在数据搬运**。FlashAttention 通过多做一点计算（recomputation）来换取巨大的内存访问节省，这是一个非常划算的交易。

右侧的 **Block Size 效果图** 进一步验证了 tiling 策略的有效性。随着 block size 从 64 增大到 512，HBM 访问量（绿色线）和运行时间（蓝色线）都单调下降。更大的 block size 意味着每次加载到 SRAM 中的数据块更大，可以在片上完成更多计算，从而减少对 HBM 的往返访问次数。但 block size 也不能无限大，因为 SRAM 的容量有限（A100 上仅约 20MB）。

论文中的原话精炼地概括了技术要点："We apply two established techniques (**tiling**, **recomputation**) to overcome the technical challenge of computing exact attention in sub-quadratic HBM accesses." 这里有两个关键词：**tiling**（分块计算）和 **recomputation**（重计算）。这两者都不是全新的发明，而是已有的成熟技术，FlashAttention 的贡献在于巧妙地将它们组合起来，解决了 attention 中 softmax 带来的特殊困难。

## Slide 51

![Slide 51](面试准备/课程记录/CS336/Lecture_05/images/page_51.png)

### 讲解

在深入 FlashAttention 的具体实现之前，我们需要先回顾 **attention 的计算流程**，理解它为什么对内存如此不友好。

Attention 的核心公式为：

```math
\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d}}\right) V
```

从计算图的角度看，这个过程包含 **3 次矩阵乘法**和**一次 softmax**：

**第一步：计算注意力分数矩阵。** 输入序列经过线性变换得到 Q（Query）和 K（Key）矩阵，然后计算 $`S = QK^T`$。如果序列长度为 $`n`$，每个 head 的维度为 $`d`$，那么 Q 和 K 的形状都是 $`n \times d`$，而得到的注意力分数矩阵 $`S`$ 的形状是 $`n \times n`$。如幻灯片中所标注的，对于多头注意力（例如 3 个 head），结果属于 $`\mathbb{R}^{3 \times n \times n}`$——这是所有 token 对之间的注意力分数的完整集合（"3 sets of all pairs of attention scores"）。

**第二步：对分数矩阵按行做 softmax 归一化。** 将 $`S`$ 通过 softmax 转换为概率分布矩阵 $`P`$，使得每一行的和为 1。这一步确保了注意力权重是合法的概率分布。

**第三步：用归一化后的权重加权 V 矩阵。** 计算 $`O = PV`$，其中 V（Value）的形状是 $`n \times d`$，最终输出也是 $`n \times d`$，代表每个 token 通过注意力机制混合（mix）了其他所有 token 的信息后的表示。

**关键的内存瓶颈**在于那个 $`n \times n`$ 的中间矩阵 $`S`$（以及 softmax 之后的 $`P`$）。在标准实现中，这个矩阵必须完整地**物化（materialize）在 HBM 中**：先算出所有的 $`QK^T`$ 值写入 HBM，再从 HBM 读出来做 softmax，结果再写回 HBM，最后再读出来与 V 相乘。对于现代大模型中常见的序列长度（$`n`$ 可达数千甚至数万），这个 $`O(n^2)`$ 大小的矩阵会消耗大量显存，并且产生巨大的 HBM 读写开销。这就是 FlashAttention 要解决的根本问题。

## Slide 52

![Slide 52](面试准备/课程记录/CS336/Lecture_05/images/page_52.png)

### 讲解

FlashAttention 的第一个核心技术是 **Tiling（分块计算）**，这也是我们在前面学习矩阵乘法优化时已经深入了解过的技术。这张幻灯片展示了 FlashAttention 论文中的 Figure 1，说明 tiling 是如何应用到 KQV 矩阵乘法上的。

左侧是我们熟悉的 **GPU 内存层次结构**金字塔：

- **SRAM**（片上缓存）：19 TB/s 带宽，但只有约 20 MB 容量
- **HBM**（高带宽显存）：1.5 TB/s 带宽，约 40 GB 容量
- **DRAM**（CPU 主存）：12.8 GB/s 带宽，超过 1 TB 容量

SRAM 的带宽是 HBM 的约 13 倍，但容量只有 HBM 的约 1/2000。这个巨大的速度差距是 tiling 策略的根本动机——我们要尽量让计算发生在 SRAM 上，而不是反复访问 HBM。

右侧的示意图展示了 FlashAttention 的 tiling 策略：不是一次性计算整个 $`QK^T`$ 矩阵，而是通过**双层循环**来分块处理。**外层循环（Outer Loop）**遍历 K 矩阵和 V 矩阵的块，将一个 block 从 HBM 拷贝到 SRAM；**内层循环（Inner Loop）**遍历 Q 矩阵的块。对于每一对 (Q block, K block)，在 SRAM 中计算出 $`QK^T`$ 的一个 **tile**（子块），然后立即用这个 tile 进行后续计算（softmax 的一部分、与 V block 的乘法），最终将结果累加到输出矩阵的对应块中，再写回 HBM。

关键的设计思想是：**$`QK^T`$ 这个 $`N \times N`$ 的大矩阵从始至终都不会被完整地存储在 HBM 中。** 它只以 tile 的形式短暂存在于 SRAM 中，计算完毕后立即被丢弃。这就是 HBM 读写量从 40.3 GB 降到 4.4 GB 的根本原因。

如果 attention 仅仅是矩阵乘法，那 tiling 就完全够用了——我们只需要像优化普通 matmul 一样分块计算即可。**但问题是：中间还有一个 softmax！** 幻灯片底部的粗体字一针见血地指出了这个挑战："This figure 1 from the paper is literally just tiling for a KQV matrix multiply.. **But what do we do about the softmax?**"

Softmax 的困难在于：$`\text{softmax}(x_i) = \frac{e^{x_i}}{\sum_j e^{x_j}}`$，分母是整行所有元素的指数和。这意味着要计算任何一个元素的 softmax 值，你需要知道**整行的所有元素**。但在 tiling 方案中，我们每次只看到一行的一个 tile（一小部分），看不到完整的行！传统的 softmax 实现要求三遍扫描（3-pass）：先遍历找最大值、再遍历求指数和、最后遍历计算输出。这与 tile-by-tile 的计算方式根本不兼容。

这就是下一张幻灯片要解决的问题：如何在只看到数据一部分的情况下，**增量地（incrementally）**计算 softmax？

## Slide 53

![Slide 53](面试准备/课程记录/CS336/Lecture_05/images/page_53.png)

### 讲解

这张幻灯片解决了上一页提出的核心挑战：**如何在 tiling 的框架下计算 softmax？** 答案来自 Milakov 和 Gimelshein 在 2018 年提出的 **Online Softmax** 算法，它通过一个精巧的 **telescoping sum（伸缩求和）** 技巧，使得 softmax 可以被分块地（tile-by-tile）增量计算。

#### 标准 Safe Softmax（Algorithm 2）——三遍扫描

首先回顾标准的 safe softmax 实现。之所以叫"safe"，是因为直接计算 $`e^{x_i}`$ 可能导致数值溢出（比如 $`x_i = 1000`$ 时 $`e^{1000}`$ 会变成 infinity），所以实际实现会先减去最大值：

```math
y_i = \frac{e^{x_i - \max_k x_k}}{\sum_{j=1}^{V} e^{x_j - \max_k x_k}}
```

所有主流深度学习框架（TensorFlow、PyTorch）都使用这个 safe 版本。其算法需要 **三遍扫描（3 passes）**：

1. **第一遍**（第 1-4 行）：遍历所有元素，找到最大值 $`m_V = \max_k x_k`$
2. **第二遍**（第 5-8 行）：再次遍历所有元素，计算分母 $`d_V = \sum_{j=1}^{V} e^{x_j - m_V}`$
3. **第三遍**（第 9-11 行）：最后遍历所有元素，计算输出 $`y_i = \frac{e^{x_i - m_V}}{d_V}`$

三遍扫描意味着数据要从 HBM 中被读取三次。对于 tiling 来说更糟糕的是，第二遍和第三遍都**依赖于**第一遍的结果（全局最大值 $`m_V`$），所以你必须先完整扫描一遍才能开始后续计算。这使得 tile-by-tile 计算看似不可能。

#### Online Softmax（Algorithm 3）——两遍扫描 + 增量更新

Online softmax 的突破性思想是：**把找最大值和计算分母这两步合并为一步**，从三遍扫描变为两遍扫描。更重要的是，合并后的第一步可以**增量式地**处理数据——每次只看一个元素（或一个 tile）就能更新状态。

算法的核心在第 3-6 行的循环中。对于每个新遇到的元素 $`x_j`$：

1. **更新最大值**：$`m_j \leftarrow \max(m_{j-1}, x_j)`$。如果新元素更大，就更新当前的最大值。

2. **更新分母**：$`d_j \leftarrow d_{j-1} \times e^{m_{j-1} - m_j} + e^{x_j - m_j}`$

这第二步就是 **telescoping sum 技巧** 的精髓，值得仔细拆解理解。设想我们已经处理了前 $`j-1`$ 个元素，此时的分母是 $`d_{j-1} = \sum_{k=1}^{j-1} e^{x_k - m_{j-1}}`$（基于旧的最大值 $`m_{j-1}`$ 计算的）。现在来了一个新元素 $`x_j`$，可能导致最大值更新为 $`m_j`$。问题是：旧的 $`d_{j-1}`$ 是用 $`m_{j-1}`$ 算的，而我们现在需要用 $`m_j`$ 来算所有东西。解决方法是**乘以一个修正因子** $`e^{m_{j-1} - m_j}`$：

```math
d_{j-1} \times e^{m_{j-1} - m_j} = \sum_{k=1}^{j-1} e^{x_k - m_{j-1}} \times e^{m_{j-1} - m_j} = \sum_{k=1}^{j-1} e^{x_k - m_j}
```

指数中的 $`m_{j-1}`$ 刚好"伸缩"抵消了（这就是"telescoping"名字的由来）！然后再加上新元素的贡献 $`e^{x_j - m_j}`$，就得到了用新最大值 $`m_j`$ 正确归一化的分母。

这个技巧之所以对 FlashAttention 至关重要，是因为它意味着 **softmax 的中间状态只需要维护两个标量**（当前最大值 $`m`$ 和当前分母 $`d`$），而不需要存储整行的所有元素。当我们 tile-by-tile 地处理 $`QK^T`$ 的一行时，每处理完一个 tile，就用这个 telescoping sum 更新 $`m`$ 和 $`d`$，处理完所有 tile 后就得到了正确的全局 softmax 分母。第二遍扫描（第 7-9 行）再用最终的 $`m_V`$ 和 $`d_V`$ 计算出每个元素的 softmax 值。

### 讲解

这张幻灯片将所有技术拼图组合在一起，展示 **FlashAttention 前向传播（forward pass）的完整计算流程**。这是整个 Part 3 的核心，来自 Dao 2023 年的 FlashAttention-2 论文。

幻灯片中的示意图展示了一个简化的例子：K 矩阵被分成两个 tile——$`K^{(1)}`$ 和 $`K^{(2)}`$（蓝色实线框，存储在 HBM 中），V 矩阵类似地分成 $`V^{(1)}`$ 和 $`V^{(2)}`$。Q 矩阵作为一个整体（也存储在 HBM 中）。橙色虚线框表示**在 SRAM 中计算、不会物化到 HBM 的中间结果**。

**计算流程如下：**

**处理第一个 tile：** 将 $`K^{(1)}`$ 的一个 block 加载到 SRAM，与 Q 计算内积得到 $`S^{(1)} = Q(K^{(1)})^T`$——这就是注意力分数矩阵的一个子块。然后**立即在 SRAM 中**计算 $`A^{(1)} = \exp(S^{(1)})`$（这就是 **fusion of the exponential operator**——将指数运算与矩阵乘法融合，避免中间结果写回 HBM）。同时计算局部归一化因子 $`l^{(1)} = \sum_i \exp(S^{(1)})_i`$，并用它计算第一个输出块：$`O^{(1)} = \frac{A^{(1)}}{l^{(1)}} \cdot V^{(1)}`$。

**处理第二个 tile：** 加载 $`K^{(2)}`$ 的 block，计算 $`S^{(2)} = Q(K^{(2)})^T`$，然后 $`A^{(2)} = \exp(S^{(2)})`$。现在需要更新归一化因子：$`l^{(2)} = l^{(1)} + \sum_i \exp(S^{(2)})_i`$。这正是 online softmax 的 telescoping sum——旧的 $`l^{(1)}`$ 加上新 tile 的贡献（为了简化，这里省略了最大值修正的细节，实际实现中 $`l^{(1)}`$ 还要乘以修正因子 $`e^{m^{(1)} - m^{(2)}}`$）。

**关键的 rescaling 步骤（图中右侧的 Output 框）：** 第二个输出块的计算公式为：

```math
O^{(2)} = \frac{l^{(1)}}{l^{(2)}} O^{(1)} + \frac{A^{(2)}}{l^{(2)}} \cdot V^{(2)}
```

这个公式蕴含了深刻的思想。$`O^{(1)}`$ 是用 $`l^{(1)}`$（只看了第一个 tile）作为分母算出的，但正确的分母应该是 $`l^{(2)}`$（看了所有 tile）。所以需要用 $`\frac{l^{(1)}}{l^{(2)}}`$ 来**重新缩放（rescale）**第一个 tile 的贡献，使其匹配正确的全局归一化。这就是图中标注的 "**Rescaling to correct denominator**"。

总结起来，FlashAttention 的前向传播融合了三个关键技术：

1. **Tile-wise computation of inner products（分块内积计算）**：$`S`$ 矩阵按 tile 计算，永远不完整地存在于 HBM 中。
2. **Fusion of the exponential operator（指数运算融合）**：$`\exp(S)`$ 在 SRAM 中紧跟着矩阵乘法立即计算，不产生额外的 HBM 往返。
3. **Tile-wise softmax via online telescoping sum（通过在线伸缩求和的分块 softmax）**：利用 online softmax 算法，每处理一个 tile 就增量更新归一化因子，处理完所有 tile 后通过 rescaling 得到精确的 softmax 结果。

关于**反向传播（backward pass）**，幻灯片提到它采用了 **recomputation（重计算）** 策略——也是 tile-by-tile 地重新计算前向传播中的中间结果（如 $`S`$ 和 $`P`$），而不是在前向传播时把它们存储下来。这看似浪费计算（增加了 GFLOPs），但节省了巨大的 HBM 存储和访问开销。回想幻灯片 50 的数据：FlashAttention 的 GFLOPs 确实略高于标准实现，但这点额外计算换来的 HBM 访问减少（40.3 GB → 4.4 GB）使得总体运行时间大幅缩短。

### 讲解

最后一张幻灯片对整个 Lecture 5 进行了全面的回顾和总结，提炼出三个核心要点（takeaways），每一个都配有对应的可视化图表，形成了从硬件趋势到编程实践的完整叙事。

#### 要点一：硬件算力在扩展，底层细节决定了什么能扩展、什么不能

右上角的图表展示了 **matmul vs non-matmul FLOPS 在不同 GPU 代际上的增长趋势**（纵轴为对数刻度的 TFLOP/S）。从 K80 到 H100，matmul 算力（橙色线）呈现了惊人的指数增长——特别是从 P100 到 V100 引入 Tensor Core 后出现了飞跃式提升，到 H100 已达约 1000 TFLOP/S。相比之下，non-matmul 算力（蓝色线）的增长要缓慢得多，到 H100 也仅约 60 TFLOP/S。

这个差距揭示了一个深刻的设计哲学：GPU 硬件的发展越来越**专注于加速矩阵乘法**，因为深度学习工作负载的核心就是矩阵运算。但这也意味着，如果你的程序中有大量的非 matmul 操作（比如逐元素运算、归约操作、数据搬运），那么硬件的算力提升将无法充分惠及你的应用。**理解底层硬件的特性，决定了你能否有效利用摩尔定律带来的红利。**

#### 要点二：当前的 GPU 计算范式强烈鼓励我们思考矩阵乘法 + 数据搬运

中间的图表展示了 **硬件 FLOPS、DRAM 带宽和互联带宽在过去 20 年的扩展趋势**。数据触目惊心：

- **HW FLOPS**：20 年增长了 60000 倍（每 2 年翻 3 倍）
- **DRAM BW**：20 年增长了 100 倍（每 2 年翻 1.6 倍）
- **Interconnect BW**：20 年增长了 30 倍（每 2 年翻 1.4 倍）

计算能力的增长远远超过了内存带宽和互联带宽的增长。这意味着随着时间推移，**内存墙（memory wall）** 问题越来越严重——GPU 的计算单元越来越快，但数据喂不进去。因此，现代 GPU 编程的核心思维方式必须围绕两件事：（1）如何把计算**重塑为矩阵乘法**以利用 Tensor Core 的澎湃算力；（2）如何**最小化数据搬运**以避免被内存带宽卡脖子。

#### 要点三：仔细思考 GPU 特性（合并访存、分块计算、算子融合）能带来巨大性能提升

右下角重新出现了 FlashAttention 的 tiling 示意图和内存层次结构金字塔，提醒我们这节课学到的三大具体优化技术：

- **Coalescing（合并访存）**：确保同一个 warp 中的线程访问连续的内存地址，使硬件能将多个小访问合并为一个大的内存事务，最大化内存带宽利用率。
- **Tiling（分块计算）**：将大的计算任务拆分为适合 SRAM 容量的小块，在片上高速缓存中完成尽可能多的计算，最小化对 HBM 的访问次数。
- **Fusion（算子融合）**：将多个连续的操作合并为一个 kernel，避免中间结果写回 HBM 再读回来的开销。

FlashAttention 就是这三种技术综合运用的典范。它没有发明全新的算法——tiling 和 recomputation 都是"established techniques"——但通过深入理解 GPU 的内存层次结构，并巧妙结合 online softmax 算法，它将标准 attention 的运行时间缩短了近 6 倍，同时将 HBM 访问量减少了约 9 倍。这个案例有力地说明了：**理解硬件、贴近硬件编程，是释放 GPU 全部潜力的关键。**
